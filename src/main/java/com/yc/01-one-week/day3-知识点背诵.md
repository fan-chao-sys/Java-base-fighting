# Day3 Java字符串底层核心知识点背诵笔记

---

## 一、String 类底层原理（必背）

### 1. 底层存储结构
- **JDK 8及之前**：底层是 `final char[] value`，用字符数组存储字符串内容。
- **JDK 9及之后**：底层改为 `final byte[] value` + `byte coder`，采用`Latin-1/UTF-16`编码优化内存占用。
- **核心本质**：`value` 数组被 `final` 修饰，数组引用不可变，且没有对外暴露修改数组的方法，因此 `String` 是**不可变类**。

### 2. 不可变特性（Immutable） ⭐
#### 底层原理
- 所有修改 `String` 的操作（如 `substring`、`replace`、`concat`、`+`拼接），都不会修改原对象，而是**创建新的 String 对象**并返回。
- 示例：`String s = "abc"; s = s + "d";` 底层会创建新的字符串对象，原 `"abc"` 对象不变。

#### 核心特性
- **线程安全**：不可变对象天然线程安全，无需同步处理。
- **缓存哈希值**：`String` 缓存了 `hashCode`，计算一次后重复使用，提升哈希表操作性能。
- **常量池复用**：支持字符串常量池，减少重复对象创建，节省内存。

## 二、字符串常量池（String Pool）底层机制

### 1. 常量池的作用
存储字符串字面量，实现相同字符串对象的复用，减少内存占用。

### 2. 两种创建方式的底层区别
#### （1）字面量创建：`String s = "abc";`
- 底层逻辑：
  1.  先检查字符串常量池中是否存在 `"abc"`。
  2.  存在：直接返回常量池中的引用。
  3.  不存在：在常量池中创建 `"abc"` 对象，再返回引用。
- 特点：**只可能创建1个对象（常量池内）**。

#### （2）new 创建：`String s = new String("abc");`
- 底层逻辑：
  1.  先检查常量池是否存在 `"abc"`，不存在则在常量池创建对象。
  2.  在堆内存中创建一个新的 `String` 对象，其内部 `value` 数组指向常量池中的 `"abc"`。
  3.  返回堆中对象的引用。
- 特点：**至少创建1个对象（堆内），常量池不存在时会创建2个对象**。

### 3. intern() 方法原理
#### 核心定义
调用 `s.intern()` 时：
- 若常量池中存在与 `s` 内容相同的字符串，直接返回常量池中的引用。
- 若不存在：
  - **JDK 6及之前**：在常量池中创建一个新对象，返回其引用。
  - **JDK 7及之后**：直接将堆中 `s` 对象的引用存入常量池，避免重复创建对象，返回该引用。

#### 作用
强制将字符串对象“入池”，复用常量池对象，减少内存占用。

## 三、String、StringBuilder、StringBuffer 对比（底层吃透）

### 1. 核心底层特性
| 类                | 底层结构       | 可变性 | 线程安全 | 性能 | 适用场景                     |
|-------------------|----------------|--------|----------|------|------------------------------|
| `String`          | `final char[]`/`final byte[]` | 不可变 | 线程安全 | 低   | 少量字符串操作、常量场景     |
| `StringBuilder`   | `char[]`/`byte[]`（无final） | 可变   | 线程不安全 | 高   | 单线程环境下大量字符串拼接   |
| `StringBuffer`    | `char[]`/`byte[]`（无final） | 可变   | 线程安全 | 较低 | 多线程环境下字符串拼接       |

---

### 2. StringBuilder 底层原理
- 底层是**可变的字符数组（JDK9+为byte数组）**，初始容量默认16，拼接字符串时直接在数组上修改，无需创建新对象。
- 当数组容量不足时，会自动扩容（扩容为原容量的2倍+2），并拷贝原数组内容到新数组。
- 无 `synchronized` 修饰，方法不保证线程安全，因此单线程下性能远高于 `StringBuffer`。

---

### 3. StringBuffer 底层原理
- 底层结构与 `StringBuilder` 相同，也是可变数组，拼接逻辑一致。
- 所有修改方法（`append`、`insert`、`delete` 等）都被 `synchronized` 修饰，保证多线程环境下的线程安全。
- 同步锁带来额外开销，因此性能比 `StringBuilder` 低。

---

## 四、高频易错点与底层避坑

### 1. 字符串拼接的底层实现
- **`+` 拼接（编译期优化）**：
  - 编译期会自动将 `String s = "a" + "b" + "c";` 优化为 `String s = "abc";`，直接复用常量池对象。
  - 若拼接包含变量（如 `String s = "a" + b + "c";`），编译期会转换为 `StringBuilder` 的 `append` 操作，拼接后调用 `toString()` 生成新对象。
- **循环拼接的性能坑**：
  - 错误写法：`for (int i=0; i<1000; i++) { s += i; }`，每次循环都会创建新的 `StringBuilder` 和 `String` 对象，时间复杂度为 O(n²)。
  - 正确写法：使用 `StringBuilder` 循环拼接，仅创建一个对象，时间复杂度为 O(n)。

### 2. new String() 的冗余创建
`new String("abc")` 会创建堆对象，若后续直接丢弃该对象，会造成内存浪费。建议优先使用字面量创建，或通过 `intern()` 方法让其复用常量池对象。

### 3. equals() 与 == 的区别
- `==`：比较对象的内存地址，判断是否为同一个对象。
- `equals()`：`String` 重写了该方法，比较字符串的内容是否相同。
- 示例：
  ```java
  String s1 = "abc";
  String s2 = new String("abc");
  System.out.println(s1 == s2);      // false（地址不同）
  System.out.println(s1.equals(s2));  // true（内容相同）
