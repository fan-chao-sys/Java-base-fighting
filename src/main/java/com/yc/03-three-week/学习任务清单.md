## Day 1：线程与进程、线程创建方式、线程状态

### 📖 基础知识点
- [ ] 进程是资源分配的最小单位，线程是 CPU 调度的最小单位；一个进程至少一个线程
- [ ] 4 种线程创建方式：继承 `Thread`、实现 `Runnable`、实现 `Callable`（有返回值+异常）、线程池
- [ ] Java 线程的 6 种状态（`Thread.State`）：NEW → RUNNABLE → BLOCKED / WAITING / TIMED_WAITING → TERMINATED
- [ ] `start()` vs `run()`：start 启动新线程，run 只是普通方法调用
- [ ] `wait()` / `notify()` / `notifyAll()` 必须在 synchronized 块内调用，否则抛 `IllegalMonitorStateException`

### ❓ 常见面试题
**Q1: `Runnable` 和 `Callable` 的区别？**
> 答：`Runnable` 的 `run()` 无返回值，不能抛受检异常；`Callable` 的 `call()` 有返回值，可抛异常。`Callable` 需配合 `FutureTask` 或线程池的 `submit()` 使用。

**Q2: 线程各状态之间怎么流转？**
> 答：NEW（创建未启动）→ `start()` → RUNNABLE（就绪/运行中）→ 等锁 → BLOCKED → `wait()` → WAITING → `sleep(n)` / `wait(n)` → TIMED_WAITING → 执行完毕 → TERMINATED。RUNNABLE 包含 ready 和 running，由 OS 调度切换。

## Day 2：线程安全本质、共享变量、原子性+可见性+有序性

### 📖 基础知识点
- [ ] 线程安全问题本质：多个线程同时操作共享可变变量，且没有同步保护
- [ ] 并发问题三要素：原子性（一个操作不可再分）、可见性（一个线程修改，其他线程立即可见）、有序性（指令重排序不乱序）
- [ ] JMM（Java 内存模型）：主内存（所有线程共享） + 工作内存（每个线程私有副本），8 种原子操作（lock/unlock/read/load/use/assign/store/write）
- [ ] happens-before 规则：程序次序、volatile、锁、线程启动/终止、传递性等 8 条规则
- [ ] 指令重排：编译器、JIT、CPU 都可能重排，`volatile` 禁止特定重排

### ❓ 常见面试题
**Q1: 什么情况下会产生线程安全问题？怎样避免？**
> 答：多个线程同时操作共享可变变量时产生。解决：① 无共享（ThreadLocal / 栈封闭）② 不可变（final）③ 同步（synchronized / Lock）④ volatile（仅保证可见性，不保证原子性）⑤ 原子类（AtomicXxx）。

## Day 3：精看 synchronized、锁升级

### 📖 基础知识点
- [ ] `synchronized` 本质是对象监视器锁（Monitor），每个对象有一个 Monitor
- [ ] 三种用法：修饰普通方法（锁 this）、修饰静态方法（锁 class 对象）、同步代码块（锁指定对象）
- [ ] 锁升级路径（JDK 6+ 优化）：无锁 → 偏向锁（记录线程 ID，同一线程重入无需 CAS）→ 轻量级锁（CAS 自旋，无实际阻塞）→ 重量级锁（自旋失败，OS 的 mutex 阻塞）
- [ ] 偏向锁撤销时机：有其他线程竞争时撤销，升级为轻量级锁
- [ ] `wait()` 释放锁并等待，`notify()` 唤醒一个等待线程（不释放锁，等同步块结束才释放）

### ❓ 常见面试题
**Q1: `synchronized` 锁升级过程是怎样的？**
> 答：① 偏向锁：第一次获取时 CAS 设置线程 ID，之后同一线程重入只检查 ID；② 轻量级锁：有竞争时，线程在栈帧中创建 Lock Record，CAS 替换 Mark Word 为指针，失败则自旋重试；③ 重量级锁：自旋超限或竞争激烈，膨胀为 monitor，未获取到锁的线程阻塞等待，由 OS 调度。

**Q2: `synchronized` 和 `ReentrantLock` 区别？**
> 答：① 关键字 vs API；② synchronized 自动释放，Lock 需 finally unlock；③ Lock 可尝试获取锁（tryLock）、可中断获取（lockInterruptibly）、可公平锁；④ Lock 可绑定多个 Condition 实现精确唤醒；⑤ JDK 6+ 后 synchronized 性能已接近 Lock。

## Day 4：volatile、JMM、happens-before、指令重排

### 📖 基础知识点
- [ ] `volatile` 两大语义：① 保证可见性（修改立即刷新到主内存，读取从主内存读）；② 禁止指令重排（内存屏障）
- [ ] `volatile` 不能保证原子性：i++ 分为"读-改-写"三步，volatile 只保证每一步可见，不保证三步是原子操作
- [ ] 内存屏障：LoadLoad / LoadStore / StoreStore / StoreLoad 四种，`volatile` 写前后插入 StoreStore 和 StoreLoad 屏障
- [ ] DCL（双重检查锁单例）：`instance = new Singleton()` 分三步（分配内存→初始化→赋值引用），2 和 3 可能重排，必须加 `volatile` 防止拿到半初始化对象

### ❓ 常见面试题
**Q1: `volatile` 为什么不能保证原子性？**
> 答：原子性要求"一个操作不可分割"，但 i++ 实际是三步：读变量 → 加 1 → 写回。volatile 只保证每个线程读到的值是最新的（可见性），不保证任何线程在这三步中间不被打断。多线程 i++ 仍会丢失更新，需要用 `synchronized` 或 `AtomicInteger`。

**Q2: DCL 单例为什么需要 volatile？**
> 答：`instance = new Singleton()` 在字节码层面不是原子的：分配内存→调用构造→赋值引用。后两步可能被指令重排，导致其他线程看到"非 null 但未初始化完"的对象。volatile 禁止重排，保证对象初始化完成后才赋值引用。

## Day 5：ReentrantLock、AQS、CAS、原子类

### 📖 基础知识点
- [ ] AQS（AbstractQueuedSynchronizer）：基于 FIFO 双向队列 + `state` 变量（CAS 修改），提供 `acquire`/`release` 模板方法
- [ ] `ReentrantLock` 基于 AQS 实现：`state=0` 未锁，`state>0` 已锁（值是重入次数）
- [ ] 公平锁 vs 非公平锁：公平锁先检查队列中有无等待线程，非公平锁直接尝试 CAS 抢锁（性能更高但可能饥饿）
- [ ] CAS（Compare And Swap）：CPU 原子指令，比较并交换，无锁算法的基石；缺点：ABA 问题、自旋循环消耗 CPU、只能保证一个变量的原子操作
- [ ] `AtomicInteger` / `AtomicReference`：基于 CAS 的原子类，`AtomicStampedReference` 可解决 ABA 问题（带版本号）
- [ ] `LongAdder`：高并发场景比 `AtomicLong` 性能更好（分段累加，减少 CAS 竞争）

### ❓ 常见面试题
**Q1: AQS 是什么？核心思想？**
> 答：AQS 是 JUC 锁和同步器的基类，核心是 `state`（同步状态）+ CLH 变体的 FIFO 等待队列。`acquire` 时 CAS 尝试修改 state，失败则入队自旋/阻塞；`release` 时修改 state 并唤醒后继节点。`ReentrantLock`、`Semaphore`、`CountDownLatch` 都是基于 AQS 实现的。

**Q2: CAS 的 ABA 问题是什么？怎么解决？**
> 答：线程 1 读到值 A，准备 CAS → 线程 2 把 A 改成 B 又改回 A → 线程 1 CAS 成功，以为值没变。解决：`AtomicStampedReference`（版本号）或 `AtomicMarkableReference`（布尔标记）。

## Day 6：线程池参数、拒绝策略、阻塞队列、ThreadLocal

### 📖 基础知识点
- [ ] 线程池七大参数：corePoolSize（核心线程数）、maximumPoolSize（最大线程数）、keepAliveTime（空闲线程存活时间）、unit（时间单位）、workQueue（阻塞队列）、threadFactory（线程工厂）、handler（拒绝策略）
- [ ] 线程池工作流程：新任务 → 核心线程有空 → 分配核心线程 → 核心线程满 → 进队列 → 队列满 → 创建新线程到 max → max 满 → 执行拒绝策略
- [ ] 4 种拒绝策略：AbortPolicy（抛异常，默认）、CallerRunsPolicy（调用者执行）、DiscardPolicy（直接丢弃）、DiscardOldestPolicy（丢弃最早任务）
- [ ] `ThreadLocal`：每个线程私有副本；底层 `Thread.threadLocals`（ThreadLocalMap），key 是弱引用，value 强引用 → 不 remove 会内存泄漏
- [ ] `CompletableFuture`：异步编排工具，链式回调（thenApply/thenAccept/thenCombine）、异常处理（exceptionally）、多任务组合（allOf/anyOf）

### ❓ 常见面试题
**Q1: 线程池为什么不建议直接用 `Executors`？**
> 答：`newFixedThreadPool` 和 `newSingleThreadExecutor` 用无界 `LinkedBlockingQueue`，可能导致 OOM；`newCachedThreadPool` 允许创建无限线程（max = Integer.MAX_VALUE），也可能 OOM。推荐用 `new ThreadPoolExecutor(...)` 自定义参数。

**Q2: `ThreadLocal` 为什么会内存泄漏？怎么避免？**
> 答：`ThreadLocalMap` 的 Entry 的 key（ThreadLocal）是弱引用，被 GC 回收后 key 变 null，但 value 是强引用且 Entry 还挂在 `Thread` 的生命周期上，造成 value 无法回收。解决：使用完一定要调 `remove()`。
