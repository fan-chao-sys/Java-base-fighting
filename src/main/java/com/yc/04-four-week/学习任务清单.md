## Day 1：运行时数据区、堆栈方法区、对象创建过程

### 📖 基础知识点
- [ ] JVM 运行时数据区 5 大块：堆（所有线程共享，存放对象实例）、方法区（元空间，类信息/常量/静态变量）、虚拟机栈（线程私有，栈帧=局部变量表+操作数栈+动态链接+返回地址）、本地方法栈（Native 方法）、程序计数器（当前执行的字节码行号）
- [ ] 堆的结构：年轻代（Eden + S0 + S1，默认 8:1:1）+ 老年代
- [ ] 对象创建流程（5 步）：① 类加载检查 → ② 分配内存（指针碰撞/空闲列表）→ ③ 初始化零值 → ④ 设置对象头（Mark Word + 类型指针）→ ⑤ 执行 `<init>`
- [ ] 对象内存布局：对象头 + 实例数据 + 对齐填充
- [ ] JDK 8 方法区变迁：永久代（PermGen）→ 元空间（Metaspace），元空间使用本地内存

### ❓ 常见面试题
**Q1: JVM 内存结构和 Java 内存模型（JMM）有什么区别？**
> 答：JVM 内存结构是物理划分（堆/栈/方法区等，描述内存区域），JMM 是逻辑规范（主内存+工作内存，描述可见性/有序性/原子性规则）。JVM 内存讲的是"数据放哪"；JMM 讲的是"多线程怎么访问共享数据"。

**Q2: 对象在内存中长什么样？**
> 答：对象头（Mark Word：哈希码/GC 年龄/锁信息 + 类型指针指向类元数据）→ 实例数据（各字段值，包括父类字段）→ 对齐填充（保证 8 字节对齐）。

## Day 2：对象分配、逃逸分析、栈上分配、TLAB

### 📖 基础知识点
- [ ] 对象分配优先级：栈上分配（逃逸分析）> TLAB（Thread Local Allocation Buffer，线程本地分配缓冲）> Eden 区 > 老年代（大对象直接进）
- [ ] 逃逸分析：分析对象作用域，未逃逸的对象可栈上分配（随栈帧销毁而回收，无需 GC）
- [ ] TLAB：每个线程在 Eden 区有独立小空间，线程内分配对象无需同步，减少 CAS 竞争
- [ ] 大对象直接进老年代：可通过 `-XX:PretenureSizeThreshold` 设置阈值
- [ ] 对象年龄：每熬过一次 Minor GC 年龄 +1，默认 15 岁进老年代（`-XX:MaxTenuringThreshold`）；动态年龄判断：同龄对象总和 > Survivor 50% 则直接晋升

### ❓ 常见面试题
**Q1: 什么情况下对象会进入老年代？**
> 答：① 大对象直接进（PretenureSizeThreshold）；② 年龄超过 MaxTenuringThreshold（默认 15）；③ 动态年龄：Survivor 中同龄对象总大小 > Survivor 空间的 50%；④ Minor GC 后 Survivor 放不下，直接进老年代（分配担保）。

**Q2: 逃逸分析做了哪些优化？**
> 答：① 栈上分配：对象分配在栈帧，GC 压力为零；② 锁消除：确定对象不会被其他线程访问，去掉同步；③ 标量替换：把对象拆成基本类型字段分别分配在栈上。

## Day 3：类加载过程、双亲委派、类加载器

### 📖 基础知识点
- [ ] 类加载 5 阶段：加载（找到 class 文件字节码 → 方法区）→ 验证（文件格式/元数据/字节码/符号引用验证）→ 准备（分配内存+赋零值，static final 变量直接赋值）→ 解析（符号引用 → 直接引用）→ 初始化（执行 `<clinit>`，static 变量赋值+static 代码块）
- [ ] 双亲委派模型：AppClassLoader → ExtClassLoader（JDK 9+ 为 PlatformClassLoader）→ BootstrapClassLoader，自底向上检查是否加载过，自顶向下尝试加载
- [ ] 双亲委派的好处：① 避免类重复加载；② 保护核心 API（如自定义 `java.lang.String` 无法加载，保证安全）
- [ ] 打破双亲委派：重写 `loadClass()` 或使用线程上下文类加载器（SPI 如 JDBC），典型场景：Tomcat 的 WebAppClassLoader

### ❓ 常见面试题
**Q1: 类加载的双亲委派模型为什么重要？**
> 答：① 防止核心类库被篡改（Bootstrap 先加载，自定义同名类不会生效）；② 保证全 JVM 中同一个类只加载一次（全限定名+类加载器确定唯一性）；③ 层次关系清晰，便于管理和隔离。

**Q2: 什么场景需要打破双亲委派？举例。**
> 答：SPI 场景如 JDBC：接口在核心库（`java.sql.Driver`，由 Bootstrap 加载），实现类在第三方 jar（如 `com.mysql.Driver`，由 AppClassLoader 加载）。Bootstrap 加载的类需要调用 AppClassLoader 加载的类，通过线程上下文类加载器（Thread Context ClassLoader）实现，Tomcat 类隔离也是典型场景。

## Day 4：垃圾回收算法、分代回收思想、可达性分析

### 📖 基础知识点
- [ ] 如何判断对象可回收：① 引用计数法（循环引用无法解决）；② 可达性分析（从 GC Roots 出发，不可达的对象判定为可回收）
- [ ] GC Roots 包括：虚拟机栈引用的对象、static 变量引用的对象、常量池引用的对象、JNI 引用的对象、同步锁持有的对象等
- [ ] 4 种引用类型：强引用（不回收）、软引用（OOM 前回收，适合缓存）、弱引用（下次 GC 必回收，如 WeakHashMap/ThreadLocal）、虚引用（无法通过它获取对象，用于跟踪回收通知）
- [ ] 垃圾回收算法：标记-清除（碎片化）→ 标记-整理（老年代，移动对象无碎片）→ 复制算法（年轻代，S0↔S1 复制，无碎片但浪费一半空间）
- [ ] 分代回收：年轻代用复制算法（对象朝生夕灭），老年代用标记-清除/整理（对象长期存活）

### ❓ 常见面试题
**Q1: 强引用、软引用、弱引用、虚引用分别用在什么场景？**
> 答：强引用：正常 new 对象，不会被 GC；软引用：内存不足时回收，适合内存敏感缓存；弱引用：下次 GC 必回收，适合 WeakHashMap（Entry 的 key）、ThreadLocal；虚引用：无法获取对象，配合 ReferenceQueue 跟踪对象回收时机（如堆外内存管理 DirectByteBuffer）。

**Q2: 年轻代和老年代分别用什么回收算法？为什么？**
> 答：年轻代用复制算法，因为对象大多"朝生夕灭"（98%），只需少量存活对象复制到 Survivor，效率高；老年代用标记-清除或标记-整理，因为对象存活率高，复制成本大且空间利用率低。

## Day 5：CMS、G1、常见 GC 日志与参数

### 📖 基础知识点
- [ ] CMS（Concurrent Mark Sweep）：老年代收集器，4 阶段：初始标记（STW，仅标 GC Roots 直连）→ 并发标记（与用户线程并行）→ 重新标记（STW，修正并发期间变化）→ 并发清除（与用户线程并行）
- [ ] CMS 缺点：① 并发阶段占用 CPU 资源；② 浮动垃圾（并发标记后产生的新垃圾不能回收）；③ 碎片化（须预留空间给浮动垃圾）→ 碎片过多触发 Full GC（Serial Old）
- [ ] G1（Garbage First）：将堆划分为多个 Region（默认 2048 个），不再物理区分年轻代/老年代；4 阶段：初始标记 → 并发标记 → 最终标记 → 筛选回收（选择回收价值最高的 Region）
- [ ] G1 优势：可预测停顿时间（`-XX:MaxGCPauseMillis`，默认 200ms）、无碎片化、增量回收
- [ ] 常见 JVM 参数：`-Xms/-Xmx`（堆初始/最大）、`-Xmn`（年轻代）、`-XX:+UseG1GC`（使用 G1）、`-XX:+PrintGCDetails`（打印 GC 日志）

### ❓ 常见面试题
**Q1: CMS 和 G1 怎么选？**
> 答：CMS 适合堆较小（≤ 4-8G）、对停顿时间有要求但不需要可预测的场景（JDK 14 已废弃）；G1 适合堆较大（≥ 6G）、需要可预测停顿时间、追求吞吐+延迟平衡的场景。JDK 9+ 默认是 G1。

**Q2: G1 的 Region 设计比传统分代好在哪？**
> 答：① 不再物理隔离年轻代/老年代，灵活调整比例；② Mixed GC 可以只回收部分老年代 Region（回收价值最高），避免全堆扫描；③ 可预测停顿时间，每次只回收预估时间内的 Region；④ 使用复制算法整理碎片化的 Region。

## Day 6：OOM 类型、内存泄漏、Full GC 排查

### 📖 基础知识点
- [ ] OOM 常见类型：堆溢出（`java.lang.OutOfMemoryError: Java heap space`）、元空间溢出（`Metaspace`）、栈溢出（`StackOverflowError`/`unable to create new native thread`）、直接内存溢出（`Direct buffer memory`）
- [ ] 内存泄漏常见原因：ThreadLocal 未 remove、静态集合不断添加、连接未关闭、Listener 未注销
- [ ] 排查工具：`jps`（查看 Java 进程）、`jstack`（线程栈）、`jmap -dump`（生成堆 dump）、`jstat -gcutil`（GC 统计）、MAT/JProfiler（分析 dump）
- [ ] Full GC 触发场景：① System.gc() 建议；② 老年代空间不足（分配担保失败）；③ 元空间不足；④ CMS GC 时 Concurrent Mode Failure（浮动垃圾过多）

### ❓ 常见面试题
**Q1: 线上频繁 Full GC 你怎么排查？**
> 答：① `jstat -gcutil <pid> 1000` 看 GC 频率和堆各区域占用；② `jmap -dump` 生成 dump 分析大对象；③ 看 GC 日志定位是常规晋升还是大对象直入；④ 调大老年代或优化代码（减少对象创建、及时释放引用）；⑤ 如果是元空间不足，检查是否大量动态生成类（如 CGLIB 代理类）。

**Q2: 什么是 Concurrent Mode Failure？**
> 答：CMS 并发清理阶段，老年代剩余空间不足以容纳 Young GC 晋升的对象或新分配的大对象。此时 CMS 会退化用 Serial Old 做一次 STW 的 Full GC，停顿时间长。
