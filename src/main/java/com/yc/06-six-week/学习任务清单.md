## Day 1：IOC、DI、BeanDefinition、BeanFactory、ApplicationContext

### 📖 基础知识点
- [ ] IOC（控制反转）：把对象的创建和管理交给 Spring 容器，而不是在代码中 new；DI（依赖注入）是 IOC 的实现方式
- [ ] DI 三种注入方式：构造器注入（推荐，不可变+不依赖反射）> setter 注入（可选依赖）> 字段注入（`@Autowired` 直接加字段上，隐藏依赖+不利于测试）
- [ ] `BeanDefinition`：Bean 的元数据定义，包含 class、scope、lazy、initMethod、destroyMethod 等
- [ ] `BeanFactory`：最底层的 IoC 容器，延迟初始化（第一次 getBean 才创建）
- [ ] `ApplicationContext`：继承 `BeanFactory`，启动时就初始化所有非懒 Bean（Eager），额外提供事件发布、国际化、环境抽象等

### ❓ 常见面试题
**Q1: `BeanFactory` 和 `ApplicationContext` 区别？**
> 答：BeanFactory 是 Spring 底层的 IoC 容器，懒加载（getBean 时才创建），功能简单；ApplicationContext 继承 BeanFactory，启动时就初始化所有单例 Bean，额外提供事件发布、国际化、资源加载、AOP 等功能。大部分场景用 ApplicationContext。

**Q2: 构造器注入为什么比字段注入好？**
> 答：① 构造器注入的依赖是 final 不可变；② 避免 NPE（依赖在构造时就校验）；③ 显式暴露依赖（类有太多依赖一眼看出来）；④ 便于单元测试（不用反射注入 mock）；⑤ 字段注入依赖 Spring 容器，脱离容器无法使用。

## Day 2：Bean 生命周期、后置处理器、初始化与销毁过程

### 📖 基础知识点
- [ ] Bean 生命周期主线：实例化（`createBeanInstance`）→ 属性填充（`populateBean`，依赖注入）→ 初始化前（`BeanPostProcessor.postProcessBeforeInitialization`）→ 初始化（`InitializingBean.afterPropertiesSet` + `@PostConstruct` + `init-method`）→ 初始化后（`BeanPostProcessor.postProcessAfterInitialization`，AOP 在此生成代理）→ 使用 → 销毁（`@PreDestroy` + `DisposableBean.destroy` + `destroy-method`）
- [ ] `BeanPostProcessor`：后置处理器，在初始化前后做回调，AOP 代理、`@Autowired` 都是通过它实现
- [ ] Bean 作用域：singleton（默认，单例）、prototype（每次获取新实例）、request/session/application（Web 环境）
- [ ] `@PostConstruct` vs `InitializingBean.afterPropertiesSet` vs `init-method` 执行顺序

### ❓ 常见面试题
**Q1: Bean 的生命周期包含哪些关键节点？**
> 答：实例化 → 属性填充 → `BeanNameAware` → `BeanFactoryAware` → `BeanPostProcessor.before` → `@PostConstruct` → `InitializingBean.afterPropertiesSet` → `init-method` → `BeanPostProcessor.after`（AOP 生成代理）→ 就绪 → `@PreDestroy` → `DisposableBean.destroy` → `destroy-method`。

**Q2: `BeanPostProcessor` 在哪个环节起作用？有什么用？**
> 答：在初始化前后分别执行 before 和 after。before 可以修改 Bean 属性（如 `@Value` 解析）；after 可以生成代理对象替代原 Bean（AOP 在此实现）。`AutowiredAnnotationBeanPostProcessor` 处理 `@Autowired` 注入。

## Day 3：AOP、动态代理、切面、通知、切点

### 📖 基础知识点
- [ ] AOP（面向切面编程）：把横切关注点（日志/事务/鉴权）和业务逻辑分离
- [ ] 核心概念：切面（Aspect=通知+切点）、通知（Advice，5 种：@Before/@After/@AfterReturning/@AfterThrowing/@Around）、切点（Pointcut，表达式匹配哪些方法）、连接点（JoinPoint，被拦截的方法）
- [ ] Spring AOP 底层：默认 JDK 动态代理（有接口），无接口用 CGLIB；Spring Boot 2.x 默认用 CGLIB（`spring.aop.proxy-target-class=true`）
- [ ] `@Transactional` 原理：通过 AOP 切面，在方法前后开启/提交/回滚事务（`TransactionInterceptor`）

### ❓ 常见面试题
**Q1: AOP 底层是怎么实现的？**
> 答：Spring AOP 基于动态代理：如果目标类实现了接口，默认用 JDK 动态代理（`Proxy.newProxyInstance`）；如果没有接口或用 `proxyTargetClass=true`，则用 CGLIB 生成子类代理。`@Transactional`、`@Cacheable` 都是通过 AOP 实现。

**Q2: `@Around`、`@Before`、`@After`、`@AfterReturning`、`@AfterThrowing` 执行顺序？**
> 答：正常流程：@Around 前 → @Before → 目标方法 → @Around 返回 → @AfterReturning → @After → @Around 后；异常流程：@Around 前 → @Before → 目标方法抛异常 → @AfterThrowing → @After → @Around 异常处理。

## Day 4：事务原理、传播行为、隔离级别、事务失效场景

### 📖 基础知识点
- [ ] Spring 事务原理：`PlatformTransactionManager` + AOP 切面，开启事务 → 执行业务 → 提交/回滚
- [ ] 7 种传播行为：REQUIRED（默认，有事务则加入，无则新建）、REQUIRES_NEW（新建事务，挂起当前）、NESTED（嵌套事务，savepoint）、SUPPORTS（有则用，无则非事务）、NOT_SUPPORTED（非事务执行，挂起当前）、NEVER（抛异常）、MANDATORY（必须已有事务）
- [ ] 事务失效 5+ 常见场景：① private 方法（AOP 基于代理，私有方法不进入代理）；② 同类内部调用（`this.method()` 不走代理）；③ 异常被 catch 未抛出（AOP 只看未捕获异常）；④ 非 `RuntimeException` 默认不回滚；⑤ `@Transactional` 加在非 public 方法；⑥ 数据库引擎不支持事务（MyISAM）

### ❓ 常见面试题
**Q1: `@Transactional` 为什么会失效？（至少说 5 种）**
> 答：① 注解在 private 方法上（CGLIB/JDK 代理只能拦截 public）；② 同类内部调用（this.method() 不走代理）；③ 异常被 try-catch 吞掉；④ rollbackFor 未配置，默认只对 RuntimeException 回滚；⑤ 数据库引擎不支持事务；⑥ 多线程中，子线程的事务不受管理；⑦ 类未注入 Spring 容器。

**Q2: REQUIRED 和 REQUIRES_NEW 的区别？**
> 答：REQUIRED：存在事务则加入，不存在则新建（传播性好）；REQUIRES_NEW：总是新建事务，挂起当前事务（完全隔离，当前事务的回滚不影响新事务）。

## Day 5：循环依赖、三级缓存、Spring MVC 请求流程

### 📖 基础知识点
- [ ] 循环依赖场景：A 依赖 B，B 依赖 A；Spring 只能解决单例 setter/字段注入的循环依赖，构造器注入的不行
- [ ] 三级缓存：一级（`singletonObjects`，完全初始化好的 Bean）→ 二级（`earlySingletonObjects`，提前暴露的半成品，未完成属性填充）→ 三级（`singletonFactories`，`ObjectFactory` 可生成早期引用或代理对象）
- [ ] 解决流程：A 实例化 → 放入三级缓存 → 填充属性 B → getBean(B) → B 实例化 → B 填充属性 A → 从三级缓存拿到 A 的早期引用 → B 初始化完成 → A 继续填充完成
- [ ] Spring MVC 请求流程：DispatcherServlet（前端控制器）→ HandlerMapping（找到 Handler）→ HandlerAdapter（执行 Handler）→ Handler（Controller）→ 返回 ModelAndView → ViewResolver（解析视图）→ 渲染返回

### ❓ 常见面试题
**Q1: Spring 是怎么解决循环依赖的？构造器循环依赖为什么不行？**
> 答：通过三级缓存：A 实例化后把 ObjectFactory 放入三级缓存 → 填充 B 时触发 B 的创建 → B 填充 A 时从三级缓存获取 A 的半成品引用 → B 完成初始化进入一级缓存 → A 用完整 B 完成初始化。构造器循环依赖不行，因为构造器注入发生在实例化阶段，此时还没有放入缓存的"半成品"可用。

**Q2: 为什么需要三级缓存？两级不行吗？**
> 答：二级缓存存的是原始对象，但 AOP 需要在初始化时生成代理对象。三级缓存存的是 `ObjectFactory`，可以先返回原始对象，等 A 初始化完成后再通过 `ObjectFactory` 生成代理对象替换。本质上是解耦了"获取半成品引用"和"是否要代理"。

## Day 6：Spring Boot 自动装配原理、设计模式

### 📖 基础知识点
- [ ] 自动装配入口：`@SpringBootApplication` → `@EnableAutoConfiguration` → `@Import(AutoConfigurationImportSelector.class)` → 读取 `META-INF/spring/org.springframework.boot.autoconfigure.AutoConfiguration.imports` 文件（Spring Boot 3+，旧版是 `spring.factories`）→ 按 `@Conditional` 条件过滤 → 加载配置类
- [ ] 核心条件注解：`@ConditionalOnClass`（类存在时才加载）、`@ConditionalOnMissingBean`（容器中没有该 Bean 才创建）、`@ConditionalOnProperty`（配置项匹配才生效）、`@ConditionalOnBean`（存在指定 Bean 才生效）
- [ ] Spring 中常用的设计模式：单例（Bean 默认 singleton）、工厂（`BeanFactory`）、代理（AOP）、模板方法（`JdbcTemplate`）、观察者（`ApplicationEvent`/`@EventListener`）、策略（`DispatcherServlet` 选择 Handler）

### ❓ 常见面试题
**Q1: Spring Boot 自动装配的核心机制是什么？**
> 答：通过 `@EnableAutoConfiguration` 导入 `AutoConfigurationImportSelector`，读取 classpath 下所有 `AutoConfiguration.imports` 文件中声明的自动配置类，再按 `@ConditionalOnXxx` 注解条件过滤，按需加载。比如 classpath 有 `DataSource` 类且有数据源配置时，才自动装配 DataSource Bean。

**Q2: 说说设计模式在 Spring 中的应用？**
> 答：单例模式（Bean 默认 singleton）、工厂模式（BeanFactory/ApplicationContext 创建管理 Bean）、代理模式（AOP 动态代理）、模板方法（JdbcTemplate/RedisTemplate 等 Template 模式）、观察者模式（ApplicationEvent 事件机制）、策略模式（DispatcherServlet 根据 HandlerMapping 选择不同 Controller）。
