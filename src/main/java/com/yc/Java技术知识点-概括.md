# Java 技术知识点总结 v2 - 面试背诵版

> 目标：把高频 Java 知识点整理成“能背、能讲、能落地”的答案库。

## 1. 集合与基础数据结构

### 基础知识
- `HashMap` 底层是数组 + 链表/红黑树，核心流程是哈希定位、冲突处理、扩容、树化。
- 集合主线：`Iterable -> Collection -> List / Set / Queue`，`Map` 是独立键值体系。
- `ArrayList` 适合随机访问，`LinkedList` 适合首尾增删，`Set` 关注去重，`Map` 关注键值映射。
- `equals` 和 `hashCode` 必须一致，否则会影响 `HashSet`、`HashMap`、`HashTable` 这类哈希结构的行为。

### 背诵答案
- **HashMap 为什么容量是 2 的幂？**  
  因为这样可以用位运算代替取模，提升性能，同时让高位扰动后的哈希值更均匀地落到桶里。
- **负载因子为什么默认是 0.75？**  
  这是空间和冲突之间的平衡点，能在扩容成本和查询效率之间取得较好的折中。
- **HashMap 什么时候树化？为什么不一开始就用红黑树？**  
  当链表长度达到阈值且数组容量足够大时会树化，因为红黑树维护成本更高，小数据量下链表更划算。
- **equals 和 hashCode 不一致会怎样？**  
  会导致对象明明相等却找不到，或者重复放入哈希容器，出现“明明同一个对象却查不出来”的问题。
- **HashSet 为什么能去重？**  
  因为它底层是 `HashMap`，元素作为 `key` 存储，重复元素会被哈希和 `equals` 判定为同一个 key。

### 真实对策
- 设计 `HashMap` key 时尽量保持不可变，避免字段变化后哈希值失效。
- 高并发场景优先考虑 `ConcurrentHashMap`，不要默认把 `HashMap` 外面包同步。
- 排序、去重、有序、并发这几个需求要先明确，再选具体集合，不要只凭习惯选。

## 2. 并发编程与多线程

### 基础知识
- 线程创建方式：`Thread`、`Runnable`、`Callable + FutureTask`、线程池。
- `synchronized` 是内置锁，`ReentrantLock` 是显式锁，`AQS` 是很多同步工具的底座。
- `volatile` 解决可见性和有序性，不保证复合操作原子性。
- `CAS` 是乐观更新基础，存在 ABA 问题。
- `ThreadLocal` 用于线程隔离上下文，但在线程池场景下要注意清理。

### 背诵答案
- **volatile 能解决什么，不能解决什么？**  
  它能保证变量对其他线程的可见性，并限制一定的指令重排，但不能保证 `count++` 这种复合操作的原子性。
- **synchronized 和 ReentrantLock 的区别是什么？**  
  `synchronized` 使用简单，JVM 会做优化；`ReentrantLock` 功能更灵活，支持可中断、可限时、公平锁和条件变量。
- **CAS 为什么会有 ABA 问题？**  
  因为值可能从 A 变成 B 又变回 A，CAS 只看当前值是否还是 A，不知道中间是否发生过变化。
- **AQS 是什么？**  
  它是 Java 并发同步器的基础框架，很多锁和同步器都基于它的队列和 state 机制实现。
- **ThreadLocal 为什么会内存泄漏？**  
  因为 `ThreadLocalMap` 的 key 是弱引用，但 value 是强引用，如果线程长期存活且不及时 `remove`，value 可能残留。

### 真实对策
- CPU 密集型线程池通常接近核数，IO 密集型可以适当放大，但必须结合队列长度、任务耗时和拒绝策略。
- 线程池不要直接无脑用 `Executors` 默认工厂，优先显式配置核心线程数、队列、拒绝策略和监控。
- 排查并发问题先看线程状态，再看 `jstack`，最后结合锁竞争、死循环、频繁 GC 定位。

## 3. Spring 核心体系

### 基础知识
- `IOC` 是控制反转，`DI` 是依赖注入。
- `BeanFactory` 是基础容器，`ApplicationContext` 是增强容器。
- Bean 生命周期包括实例化、属性注入、初始化、销毁。
- `AOP` 通常通过代理实现，JDK 动态代理面向接口，CGLIB 基于子类。
- 事务本质上也是通过代理和拦截器增强实现的。

### 背诵答案
- **Spring 为什么能自动装配 Bean？**  
  因为它会扫描配置和组件，通过条件判断、Bean 定义注册和依赖注入，自动把符合条件的对象装配进容器。
- **Spring 如何解决循环依赖？**  
  主要依赖单例 Bean 的三级缓存和提前暴露引用，前提通常是 setter 注入或字段注入，构造器循环依赖更难处理。
- **@Transactional 为什么会失效？**  
  常见原因有自调用、方法不是 `public`、异常被吞掉、抛出的异常类型不触发回滚、Bean 没被 Spring 管理。
- **AOP 自调用为什么不生效？**  
  因为代理只拦截外部通过代理对象发起的调用，类内部直接调用自己方法不会经过代理。
- **JDK 动态代理和 CGLIB 的区别是什么？**  
  JDK 代理基于接口，生成接口实现类；CGLIB 基于继承，生成目标类子类。

### 真实对策
- 事务方法尽量放在独立的 `public` 方法里，避免内部自调用。
- 不要在事务方法里随意 `catch` 后吞掉异常，需要回滚时要明确配置 `rollbackFor`。
- 循环依赖优先从设计上拆解，而不是仅靠框架技巧硬扛。

## 4. SpringBoot 与 SpringCloud

### 基础知识
- `SpringBoot` 的核心是约定优于配置、自动配置、Starter、内嵌容器、外部化配置。
- `SpringCloud` 建立在 `SpringBoot` 上，解决微服务治理问题。
- 常见治理组件：注册中心、配置中心、网关、限流熔断、远程调用、链路追踪。

### 背诵答案
- **SpringBoot 自动配置是怎么生效的？**  
  它通过条件注解、自动配置类导入和依赖判断，在启动时按条件创建 Bean，减少手工配置。
- **@SpringBootApplication 是什么？**  
  它是组合注解，通常包含组件扫描、自动配置和配置入口能力。
- **Starter 的作用是什么？**  
  它把一类场景所需的依赖打包成统一入口，降低接入成本，减少版本和配置冲突。
- **SpringCloud 为什么基于 SpringBoot？**  
  因为 SpringBoot 已经提供了统一的启动、配置和自动装配能力，SpringCloud 只需要在这个基础上做分布式治理。
- **注册中心、配置中心、网关分别做什么？**  
  注册中心负责服务发现，配置中心负责集中配置管理，网关负责统一入口、路由和鉴权。

### 真实对策
- 如果只是普通单体系统，不要为了“微服务”而微服务，先看团队规模和拆分边界。
- 微服务架构必须配套监控、日志、链路追踪、灰度和降级，否则故障定位会很痛苦。

## 5. Redis

### 基础知识
- 常见数据结构：`String`、`Hash`、`List`、`Set`、`ZSet`、`Bitmap`、`HyperLogLog`、`GEO`、`Stream`。
- 持久化方式：`RDB`、`AOF`、混合持久化。
- 高可用体系：主从复制、哨兵、Cluster。
- 缓存问题三件套：穿透、击穿、雪崩。
- 分布式锁必须考虑原子加锁、唯一标识、过期时间、释放校验和必要续期。

### 背诵答案
- **Redis 为什么快？**  
  因为它基于内存，数据结构简单，单线程模型减少了锁竞争，同时借助 IO 多路复用提升吞吐。
- **SET NX PX 为什么能做分布式锁？**  
  因为它能原子地完成“只有不存在时才加锁，并设置过期时间”，避免死锁和重复加锁。
- **为什么 Redis 需要 Lua？**  
  因为 Lua 脚本在 Redis 内是原子串行执行的，适合把多步操作合并成一个原子事务逻辑。
- **什么是缓存穿透、击穿、雪崩？**  
  穿透是查不存在的数据，击穿是热点 key 失效瞬间被大量请求打到数据库，雪崩是大量 key 同时失效。
- **热 key、大 key 怎么处理？**  
  热 key 要预热、拆分、做二级缓存或异步刷新；大 key 要拆结构，避免慢命令和阻塞主线程。

### 真实对策
- 分布式锁至少要做到“加锁原子、value 唯一、解锁校验身份、支持过期恢复”。
- 穿透用布隆过滤器或空值缓存，击穿用互斥锁或逻辑过期，雪崩用随机 TTL 和限流降级。

## 6. MQ

### 基础知识
- MQ 的核心价值：异步、解耦、削峰填谷、广播、最终一致性。
- 角色链路：Producer、Broker、Consumer、Topic / Queue。
- 常见问题：消息丢失、重复消费、顺序消费、积压、死信、事务消息。
- 常见产品：Kafka、RabbitMQ、RocketMQ。

### 背诵答案
- **为什么要用 MQ？**  
  因为它能把同步链路拆成异步链路，降低接口响应时间，同时削峰和解耦上下游系统。
- **消息为什么会丢？**  
  可能丢在生产端、Broker 持久化、网络传输、消费确认等多个环节。
- **为什么消费者必须做幂等？**  
  因为 MQ 常见投递语义是至少一次，重复消息是正常现象，消费者必须能重复执行而不出错。
- **怎么保证顺序消息？**  
  通常按业务 key 分区或分队列，并确保同一分区单线程顺序消费。
- **什么是事务消息？**  
  它用于协调本地事务和消息投递，解决跨系统最终一致性问题。

### 真实对策
- 可靠性一般要靠“本地事务 + 消息确认 + 重试 + 补偿 + 死信”组合解决。
- 积压时先看生产消费速率差，再决定扩容消费者、拆 Topic、优化业务逻辑或临时降级。

## 7. MySQL

### 基础知识
- InnoDB 是主流引擎，支持事务、MVCC、行锁、聚簇索引。
- 索引类型：主键、唯一、普通、联合、覆盖索引。
- 联合索引遵循最左前缀原则。
- 隔离级别：RU、RC、RR、Serializable。
- `EXPLAIN` 是 SQL 优化第一入口。

### 背诵答案
- **为什么 MySQL 用 B+Tree 而不是红黑树或哈希？**  
  因为 B+Tree 更适合磁盘存储和范围查询，层高低、IO 次数少，且叶子节点链表便于顺序扫描。
- **聚簇索引和二级索引有什么区别？**  
  聚簇索引叶子节点存整行数据，二级索引叶子节点存主键值，二级索引命中后通常需要回表。
- **什么是回表和覆盖索引？**  
  回表是先命中索引再回主键索引取整行；覆盖索引是查询字段都在索引里，不需要回表。
- **MVCC 是什么？**  
  它通过 ReadView 和 undo log 实现快照读，提升读写并发，避免大量读写互斥。
- **Next-Key Lock 解决什么问题？**  
  它结合记录锁和间隙锁，主要用于防止幻读。

### 真实对策
- 索引设计先看查询条件、排序条件和覆盖字段，再考虑写入成本。
- 避免在索引列上做函数、隐式转换和不必要的范围破坏。
- 深分页尽量改成延迟关联、游标翻页或按主键范围翻页。
- 排查慢 SQL 先看慢日志，再看 `EXPLAIN`，最后结合索引和回表成本优化。

## 8. ClickHouse / MongoDB 选型补充

### 基础知识
- ClickHouse 更适合分析型查询、聚合统计、报表场景。
- MongoDB 更适合结构变化快、文档模型明显、字段不固定的场景。
- MySQL 更适合事务型读写。

### 背诵答案
- **什么场景该选 ClickHouse？**  
  当业务以大规模分析、聚合、报表、OLAP 查询为主时，ClickHouse 通常更合适。
- **什么场景该选 MongoDB？**  
  当业务结构变化快、文档模型明显、字段不固定时，MongoDB 更灵活。
- **为什么不能把所有分析查询都压在 MySQL 上？**  
  因为会拖慢核心事务库，影响线上写入和主业务响应。

### 真实对策
- 交易库和分析库分离，避免报表查询拖垮主业务库。
- 文档结构频繁变化时再考虑 MongoDB，同时评估索引和查询复杂度。

## 9. Linux 常见问题定位

### 基础知识
- 常用命令：`top`、`ps`、`grep`、`tail -f`、`ss`、`df`、`du`、`free`、`vmstat`、`iostat`。
- Java 专项命令：`jps`、`jstack`、`jmap`、`jstat`、`jcmd`。
- 排障主线：现象 -> 指标 -> 进程 -> 线程 -> 日志 -> 根因。

### 背诵答案
- **CPU 100% 怎么查？**  
  先用 `top` 找高 CPU 进程，再用 `top -Hp` 找具体线程，最后用 `jstack` 对照线程栈判断是否死循环、锁竞争或频繁 GC。
- **内存持续增长怎么查？**  
  先区分系统内存和 Java 堆，再看 `free`、`top`、`jmap` 和堆转储，判断是泄漏还是正常增长。
- **磁盘满了怎么查？**  
  先用 `df` 看哪个分区满，再用 `du` 找大目录，最后定位日志、转储或临时文件。
- **端口不通怎么查？**  
  先看进程是否监听，再用 `ss`、`curl`、`telnet`、`ping` 逐层排查网络和应用。
- **接口超时怎么定位？**  
  先分辨是入口、应用、数据库还是外部依赖慢，再看日志、线程、SQL 和链路耗时。

### 真实对策
- 线上排障先保守采证，再做调整，避免二次事故。
- 先判断是系统问题还是 Java 问题，不要一上来就改代码。

## 10. 自定义注解与反射

### 基础知识
- 注解本质是元数据，常配合反射、AOP、编译期处理器使用。
- 重点看 `Retention`、`Target`、`Inherited`、`Documented`。
- 反射允许在运行期读取类、字段、方法、构造器、注解信息。

### 背诵答案
- **注解是怎么生效的？**  
  注解本身只是元数据，真正生效通常依赖框架在运行时通过反射读取，或者在编译期通过处理器生成代码。
- **反射为什么慢？**  
  因为它多了一层元信息查找、权限校验和动态调用开销，且不容易被 JIT 充分优化。
- **为什么很多框架离不开注解和反射？**  
  因为它们能把配置和行为从硬编码中解耦，让框架在运行时按元数据自动装配和增强。

### 真实对策
- 热点路径少用高频反射，必要时缓存 `Method`、`Field`、`Constructor`。
- 如果只是固定映射，优先代码生成或编译期处理，不要把反射当万能方案。

## 11. 微服务基础组件

### 基础知识
- Nacos：注册发现 + 配置中心。
- Gateway：统一入口、路由、鉴权、转发。
- Sentinel：限流、熔断、降级、流量保护。
- Seata：跨服务事务治理。
- Docker：统一运行环境，提高部署一致性。
- Nginx：反向代理、负载均衡、静态资源、入口转发。

### 背诵答案
- **为什么要服务注册发现？**  
  因为服务实例会动态变化，手写地址很难维护，注册发现可以让服务间调用更灵活。
- **Gateway 和 Nginx 的区别是什么？**  
  Nginx 更偏基础流量入口和反向代理，Gateway 更偏业务网关能力，如路由、鉴权、灰度和链路治理。
- **Sentinel 和 MQ 限流有什么不同？**  
  Sentinel 是在线流量保护，MQ 更多是异步削峰，两者解决的问题不同。
- **Seata 为什么不能滥用？**  
  因为分布式事务会增加系统复杂度和性能开销，不适合所有业务场景。

### 真实对策
- 微服务组件要配套监控、日志、链路追踪一起上。
- 能用简单方案解决的，优先简单方案，不要一开始就堆复杂分布式事务。

## 12. 复习方法

### 背诵建议
- 每个知识点都按“是什么、为什么、怎么实现、怎么选型、怎么排障”五步记。
- 面试回答优先先给结论，再给原理，最后补场景和代价。
- 遇到线上问题，先观察日志和指标，再定位线程、SQL、缓存或网络。

### 结论
- 这份笔记的核心不是背定义，而是能把知识点讲成“原理 + 场景 + 解决方案”。
- 真正能过面试的答案，通常不是一句话，而是“概念说清、边界说清、坑点说清、对策说清”。
