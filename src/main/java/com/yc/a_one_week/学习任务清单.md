## Day 1：基本数据类型、包装类、== 与 equals、hashCode

### 📖 基础知识点
- [ ] Java 8 种基本类型（byte/short/int/long/float/double/char/boolean）及对应包装类
- [ ] 自动装箱 `Integer i = 1` 与拆箱 `int j = i` 的触发时机，`Integer` 缓存池范围（-128 ~ 127）
- [ ] `==` 比较引用地址 vs `equals` 默认行为（Object 里就是 `==`）
- [ ] `hashCode` 约定：equals 相等的两个对象，hashCode 必须相等；反之不要求
- [ ] Java 是严格的值传递：方法传参时拷贝一份副本，引用类型拷贝的是引用地址的值

### ❓ 常见面试题
**Q1: Java 是值传递还是引用传递？举例说明。**
> 答：值传递。基本类型传的是值的副本；引用类型传的是"引用的地址值"的副本，所以方法内可以修改对象内容，但不能让外部引用指向新对象。例如 `swap(Integer a, Integer b)` 无法真正交换。

**Q2: `Integer` 用 `==` 比较为什么 100 是 true，200 是 false？**
> 答：`Integer` 内部有缓存池 `IntegerCache`，默认缓存 -128 ~ 127。在这个范围内，装箱直接返回缓存对象，同一个地址 → `==` 为 true。200 超出范围，每次装箱都会 new 新对象。

**Q3: 为什么重写 `equals` 必须重写 `hashCode`？**
> 答：`HashMap` 等容器先用 `hashCode` 定位桶，再用 `equals` 判等。如果只重写 `equals` 不重写 `hashCode`，两个 equals 相等的对象可能落在不同桶里，导致容器行为违反规约（contains 返回 false、重复 key 等）。

## Day 2：面向对象三大特性、重载重写、接口与抽象类、final/static

### 📖 基础知识点
- [ ] 封装：隐藏内部状态，暴露受控接口
- [ ] 继承：子类拥有父类的属性和方法（除 private），Java 单继承
- [ ] 多态：父类引用指向子类对象 + 方法动态绑定（编译看左边，运行看右边）
- [ ] 重载（Overload）：同一类中方法名相同、参数列表不同，编译期绑定；重写（Override）：子类覆盖父类方法，运行期绑定
- [ ] `final`：修饰类不可继承、修饰方法不可重写、修饰变量不可修改；`static`：属于类级别，所有实例共享
- [ ] 接口 vs 抽象类：接口强调"能做什么"（行为契约，Java 8+ 可有 default 方法），抽象类强调"是什么"（is-a，可含成员变量和构造器）

### ❓ 常见面试题
**Q1: 接口和抽象类怎么选？**
> 答：接口定义行为契约，支持多实现，适合解耦（如 `Runnable`、`Comparable`）；抽象类抽取公共代码和状态，适合 is-a 关系（如 `HttpServlet`）。Java 8 后接口有了 default 方法，区别变小，但抽象类仍有成员变量和构造器的优势。

**Q2: `final` 修饰类、方法、变量分别表示什么？**
> 答：修饰类 → 不可被继承（如 `String`）；修饰方法 → 不可被子类重写；修饰变量 → 基本类型值不可变，引用类型引用不可变（但对象内容可变）。

## Day 3：String、字符串常量池、StringBuilder、StringBuffer

### 📖 基础知识点
- [ ] `String` 底层是 `final char[]`（JDK 9+ 为 `final byte[]`），不可变
- [ ] 字符串常量池：`String s = "abc"` 会从常量池取，没有则创建；`new String("abc")` 会在堆创建对象
- [ ] `intern()`：将字符串放入常量池并返回池中引用
- [ ] `StringBuilder`：可变，线程不安全，性能好，单线程拼接首选
- [ ] `StringBuffer`：可变，方法都用 `synchronized` 修饰，线程安全，多线程拼接用

### ❓ 常见面试题
**Q1: `String` 为什么设计成不可变？**
> 答：① 安全：String 广泛用于参数传递，不可变防止被篡改；② 哈希缓存：hash 值可缓存，HashMap 的 key 常用 String；③ 字符串常量池复用，节省内存；④ 线程安全：不可变对象天然线程安全。⭐⭐

**Q2: `String`、`StringBuilder`、`StringBuffer` 区别？**
> 答：`String` 不可变，拼接会产生新对象；      
  `StringBuilder` 可变、线程不安全、性能最高；
  `StringBuffer` 可变、线程安全（synchronized）、性能次之。
单线程拼接用 StringBuilder，多线程用 StringBuffer或直接用 String.concat/+

## Day 4：异常体系、try-catch-finally、throw、throws

### 📖 基础知识点
- [ ] Throwable 分两支：Error（JVM 级错误，不处理）和 Exception
- [ ] Exception 分两类：受检异常（Checked，编译期强制处理，如 `IOException`）和非受检异常（Unchecked，`RuntimeException` 及其子类，如 `NullPointerException`）
- [ ] `try-catch-finally` 执行顺序：try 正常 → finally 一定执行；try 中 return → 仍先执行 finally 再 return；finally 中有 return 会覆盖 try 的 return
- [ ] `throw`：在方法内抛出异常对象；`throws`：在方法签名上声明可能抛出的受检异常
- [ ] try-with-resources（JDK 7+）：实现了 `AutoCloseable` 的资源会自动 close，无需手动 finally

### ❓ 常见面试题
**Q1: Java 异常为什么分受检和非受检？**
> 答：受检异常（Checked）代表"调用者可以处理的外部异常"，如文件不存在、网络中断，编译器强制处理让 API 更可靠；非受检异常（Unchecked）代表"程序 bug"，如空指针、越界，调用者无法合理处理，应通过代码修复而非捕获。

**Q2: finally 里 return 会怎样？**
> 答：finally 的 return 会覆盖 try 的 return，且 try 中抛出的异常会被吞掉。这是坑，应避免在 finally 中写 return。

## Day 5：泛型、类型擦除、通配符、PECS 原则

### 📖 基础知识点
- [ ] 泛型的作用：编译期类型检查，消除强制类型转换，写出更安全的代码
- [ ] 类型擦除：泛型信息在编译后会被擦除，字节码中 `<T>` 变为 `Object`（无界）或上界类型（有界 `<T extends Number>` 变为 `Number`）
- [ ] 通配符：`?` 表示未知类型；`? extends T` 上界（生产者，只能读不能写）；`? super T` 下界（消费者，只能写不能读）
- [ ] PECS 原则：Producer Extends, Consumer Super。从容器读 → `? extends T`；往容器写 → `? super T`
- [ ] 不能 `new T()`、不能 `new T[]`、不能 `instanceof T`（擦除后 T 不存在于运行时）

### ❓ 常见面试题
**Q1: 什么是类型擦除？**
> 答：Java 泛型通过编译期类型擦除实现。编译器将 `List<String>` 和 `List<Integer>` 都擦除为 `List<Object>`，字节码中无泛型信息。好处是兼容旧版本 JVM；缺点是无法在运行时获取泛型类型（但可通过反射获取方法签名中的泛型信息）。

**Q2: `List<?>`、`List<? extends Number>`、`List<? super Integer>` 各怎么用？**
> 答：`List<?>` 只能读（取出是 Object），不能写（null 除外）；`List<? extends Number>` 只能读（取出是 Number），不能写；`List<? super Integer>` 可以写 Integer，读取是 Object。能用 PECS 记住。

## Day 6：反射、注解、JDK 动态代理、CGLIB

### 📖 基础知识点
- [ ] 反射：运行时获取类的信息（Class、Field、Method、Constructor），动态创建对象和调用方法
- [ ] 反射的入口：`Class.forName("全限定名")` / `obj.getClass()` / `类名.class`
- [ ] 注解：元数据标记，`@Retention` 控制生命周期（SOURCE / CLASS / RUNTIME），`@Target` 控制作用位置
- [ ] JDK 动态代理：基于接口，`Proxy.newProxyInstance()` + `InvocationHandler`，运行时生成代理类
- [ ] CGLIB：基于继承，通过 ASM 操作字节码生成子类，不需要接口，但不能代理 final 类和方法

### ❓ 常见面试题
**Q1: JDK 动态代理和 CGLIB 区别？**
> 答：JDK 动态代理基于接口，通过 `Proxy` + `InvocationHandler` 生成代理类，只能代理实现了接口的类；CGLIB 基于继承，通过字节码技术生成目标类的子类，不需要接口，但不能代理 final 类/方法。Spring AOP 默认用 JDK 动态代理，如果目标类没实现接口则用 CGLIB。

**Q2: 反射为什么慢？慢在哪里？**
> 答：① 需要 JNI 操作和安全性检查；② 需要查找类的元数据；③ 每次调用都走动态解析而非内联优化；④ 无法被 JIT 编译器充分优化（逃逸分析等失效）。不过现代 JVM 已经大幅优化了反射性能，setAccessible(true) 也能减少权限检查开销。
