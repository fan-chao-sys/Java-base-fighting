# HashMap 面试核心背诵笔记（JDK 8+）
---

## 一、底层结构
- **核心结构**：`Node<K,V>[] table`，采用 **数组 + 链表 + 红黑树** 组合实现。
- **结构分工**：
  - 数组：作为主结构，存放桶的头节点，实现快速定位。
  - 链表：解决哈希冲突，存放同一桶下的元素。
  - 红黑树：当链表过长时自动转换，优化查询效率。

---

## 二、put 流程（面试必背）
```text
① 计算hash → ② 定位桶位置 → ③ 桶为空直接插入 → ④ key存在则覆盖 → ⑤ key不存在则插入链表/红黑树 → ⑥ 链表过长则树化 → ⑦ 超过阈值则扩容
```
1.  **计算 hash**：
    `(h = key.hashCode()) ^ (h >>> 16)`，高低位混合扰动，减少哈希冲突。
2.  **定位桶位置**：
    `i = (n - 1) & hash`，利用数组长度为2的幂，高效取模。
3.  **桶为空**：直接插入元素。
4.  **桶不为空**：
  - key 相同（`hash` 相同且 `equals` 为 `true`）：覆盖旧值。
  - key 不同：判断结构类型。
    - 链表：尾插法插入节点（JDK 8+）。
    - 红黑树：按红黑树规则插入。
5.  **树化条件**：链表长度 ≥ 8 且 数组长度 ≥ 64。
6.  **扩容触发**：插入后 `size > threshold`（容量 × 负载因子0.75），触发扩容。

## 三、哈希冲突解决
| JDK 版本 | 冲突解决方式 | 关键变化 | 优化点 |
| :--- | :--- | :--- | :--- |
| 7 | 数组 + 链表 + 头插法 | 头插法 | 无红黑树优化，长链查询慢 |
| 8+ | 数组 + 链表/红黑树 + 尾插法 | 尾插法 + 树化 | 避免扩容死循环，查询从O(n)→O(logn) |

- **核心考点**：
  - 头插法在多线程扩容时易形成环形链表，导致死循环；尾插法避免了该问题。
  - 红黑树的引入是为了解决链表过长导致的性能下降问题。

---

## 四、扩容机制
- **触发条件**：元素数量 `size > threshold`（`threshold = 容量 × 0.75`）。
- **扩容操作**：
  1.  容量翻倍（新容量为旧容量的2倍，保持为2的幂）。
  2.  遍历旧数组，将每个桶中的元素重新分配到新数组。
  3.  元素新位置：原位置 或 原位置 + 旧容量（利用扩容后二进制高位的变化）。
- **核心考点**：
  - 扩容是 `HashMap` 最耗性能的操作，因为需要重新计算哈希并复制元素。
  - 初始容量设置不合理会导致频繁扩容，影响性能。

## 五、高频面试题速记
1.  **HashMap 的底层结构？**
    JDK 8+ 是 **数组 + 链表 + 红黑树**，JDK 7 是 **数组 + 链表**。
2.  **put 流程的关键步骤？**
    计算 hash → 定位桶 → 处理冲突 → 树化 → 扩容。
3.  **为什么用尾插法？**
    避免多线程扩容时出现环形链表，导致死循环。
4.  **为什么要树化？**
    当链表过长时，查询效率从 O(n) 优化为 O(logn)。
5.  **扩容时元素位置怎么变？**
    要么留在原位置，要么移动到 `原位置 + 旧容量` 的位置。
6.  **负载因子为什么是0.75？**
    平衡空间利用率和哈希冲突概率，减少扩容次数。
