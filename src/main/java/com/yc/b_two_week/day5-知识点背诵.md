## 一、JDK 1.7 vs JDK 1.8 HashMap 核心差异（面试重中之重）

### 1.1 关键差异总结表

| 对比维度 | JDK 1.7 | JDK 1.8 |
|---|---|---|
| **数据结构** | 数组 + 单向链表 | 数组 + 链表 + 红黑树 |
| **插入方式** | **头插法**（新节点插在链表头部） | **尾插法**（新节点插在链表尾部） |
| **hash 计算** | 多次扰动（4次位运算 + 5次异或） | 简化：`h ^ (h >>> 16)`（一次右移 + 一次异或） |
| **扩容条件** | size > threshold 且 当前桶非空 | size > threshold（去掉了桶非空的判断） |
| **扩容后迁移** | 头插法倒序重建链表 | 高位/低位 分流（保持原序，不反转） |
| **死循环风险** | **有**（resize 头插法可能形成环形链表） | **无**（尾插法不会反转链表顺序） |

### 1.2 数据结构差异

```
JDK 1.7:                     JDK 1.8:
table[0] → entry → entry    table[0] → Node → Node → ...    (链表)
table[1] → entry            table[1] → TreeNode ↔ TreeNode   (红黑树)
table[2] → entry → entry    table[2] → Node
...                         ...
```

**1.7 缺点**：所有元素都在链表上，极端情况下退化成 O(n) 查询。

**1.8 优化**：链表 ≥ 8 时转为红黑树，查询从 O(n) → O(logn)，是质变级优化。

### 1.3 插入方式差异

#### 头插法（JDK 1.7）为什么危险？

```java
// JDK 1.7 transfer() 扩容迁移核心逻辑（简化）
void transfer(Entry[] newTable) {
    for (Entry<K,V> e : table) {
        while (null != e) {
            Entry<K,V> next = e.next;      // 1. 保存后继
            int i = indexFor(e.hash, newCapacity);
            e.next = newTable[i];           // 2. e 的 next 指向新桶的头
            newTable[i] = e;                // 3. e 成为新桶的头
            e = next;                        // 4. 继续处理下一个
        }
    }
}
```

**死循环成因**：多线程并发 resize 时，线程 A 和线程 B 同时操作，由于头插法**反转了链表顺序**（遍历顺序 A→B→C，迁移后变成 C→B→A），两个线程相互干扰，可能使某个节点的 next 指回其前驱，形成**环形链表**。后续调用 `get()` 查询到该桶时，遍历环形链表进入**死循环**，CPU 100%。

#### 尾插法（JDK 1.8）为什么安全？

```java
// JDK 1.8 resize() 迁移核心逻辑
// 维护高低位两条链表，保持原顺序
Node<K,V> loHead = null, loTail = null;  // 原位链表（头+尾指针）
Node<K,V> hiHead = null, hiTail = null;  // 高位链表（头+尾指针）
Node<K,V> next;
do {
    next = e.next;
    if ((e.hash & oldCap) == 0) {
        // 送进原位链表尾部
        if (loTail == null) loHead = e;
        else loTail.next = e;
        loTail = e;
    } else {
        // 送进高位链表尾部
        if (hiTail == null) hiHead = e;
        else hiTail.next = e;
        hiTail = e;
    }
} while ((e = next) != null);
```

**为什么安全**：
1. 尾插法保持元素**原有插入顺序**，不反转链表
2. 高低位分流直接定位新桶，一次遍历完成
3. 不会形成环形链表

### 1.4 hash 计算差异

| | JDK 1.7 | JDK 1.8 |
|---|---|---|
| 公式 | `h ^= (h >>> 20) ^ (h >>> 12); h ^ (h >>> 7) ^ (h >>> 4)` | `(h = key.hashCode()) ^ (h >>> 16)` |
| 扰动次数 | 4 次位运算 + 5 次异或 | 1 次位运算 + 1 次异或 |
| 设计理念 | 重扰动，试图打散低位差异 | 轻扰动，高低位异或足够（配合红黑树兜底） |

**1.8 简化的底气**：引入红黑树后，即使有哈希碰撞导致长链，也能维持 O(logn) 查询。不再需要过度优化哈希函数本身。

## 二、红黑树化条件（底层吃透）⭐⭐⭐

### 2.1 树化的两个必要条件

```java
// 条件1: 链表长度 ≥ 8  →  在 putVal 中判断
if (binCount >= TREEIFY_THRESHOLD - 1)  // TREEIFY_THRESHOLD = 8
    treeifyBin(tab, hash);

// 条件2: 数组长度 ≥ 64  →  在 treeifyBin 中再判断
if (tab == null || (n = tab.length) < MIN_TREEIFY_CAPACITY)  // 64
    resize();   // 数组太小 → 扩容而不是树化
```

**两个条件缺一不可**：
- 链表 ≥ 8 节点 → 触发树化检查
- 数组 ≥ 64 → 真正树化；否则只扩容

### 2.2 为什么阈值是 8？

**官方依据 — 泊松分布（Poisson Distribution）**：

在 loadFactor=0.75 的前提下，HashMap 中一个桶内节点数的概率分布服从均值 λ=0.5 的泊松分布：

| 桶中节点数 | 概率 |
|---|---|
| 0 | 0.60653066 |
| 1 | 0.30326533 |
| 2 | 0.07581633 |
| 3 | 0.01263606 |
| 4 | 0.00157952 |
| 5 | 0.00015795 |
| 6 | 0.00001316 |
| 7 | 0.00000094 |
| **8** | **0.00000006**（< 千万分之一） |

- 链表达到 8 个节点的概率 **不到千万分之一**
- 如果真的出现 8 个节点，说明哈希函数异常（或有人恶意构造碰撞），此时需要红黑树兜底
- 选择 8 是在"避免过早树化（浪费空间）"和"避免过晚树化（查询退化严重）"之间精打细算的平衡

### 2.3 为什么数组小了先扩容不树化？

数组容量小时，元素集中是**正常现象**（桶少装得多）。扩容能让节点重新散列到更多桶，自然缩短链表长度。此时树化会引入 TreeNode 的额外开销（每个 TreeNode 约是 Node 的 2 倍），得不偿失。

## 三、红黑树退化条件

### 3.1 退化阈值

```java
static final int UNTREEIFY_THRESHOLD = 6;
```

- 树节点 ≤ 6 时 → 红黑树退化为链表
- **为什么是 6 而不是 8？** → 中间留 7 这个缓冲区

### 3.2 缓冲区的设计意义

```
链表 ←── 6 ── 7 ── 8 ──→ 红黑树
       退化      缓冲      树化
```

如果阈值设成同一个数（如都用 8），在 8 附近频繁增删元素时，会触发**频繁的树化 ↔ 退化转换**，每次都涉及 TreeNode ↔ Node 的 O(n) 转换开销。6/8 之间的 7 充当缓冲，避免了持续抖动。

### 3.3 触发退化的两种场景

1. **resize 后的迁移**：扩容重定位桶时，如果桶元素 ≤ 6，不维护树结构，直接建链表
2. **remove 后检查**：删除节点后，如果树结构太稀疏（根节点为 null / 左右子为空等），调用 `untreeify()` 退化为链表

## 四、loadFactor 负载因子深度解析

### 4.1 什么是 loadFactor？

```java
threshold = capacity * loadFactor;
// 默认: 16 * 0.75 = 12
```

- `threshold` 是扩容阈值，当 HashMap 中的键值对数量 > threshold 时触发扩容
- loadFactor 控制"装多满才扩容"

### 4.2 0.75 是怎么定出来的？

| loadFactor | 效果 | 场景 |
|---|---|---|
| **1.0** | 空间利用率最高，但碰撞严重，查询变慢 | 内存极度紧张 |
| **0.5** | 碰撞极少，查询快，但一半空间浪费 | 对时间和正确性极度敏感 |
| **0.75** | **时间与空间的平衡折中点** | **通用默认** |

**0.75 的数学依据**：基于泊松分布模型，在 loadFactor=0.75 时，桶中节点数的概率分布整体处于最优平滑状态，k=8 的概率仅 0.00000006。

> 一般不需要修改。如果预期存入大量元素，可以在构造时预设容量：`new HashMap((int)(预计元素数 / 0.75) + 1)`，避免反复扩容。

## 五、JDK 1.7 死循环完整推演（面试加分项）

### 5.1 死循环产生前提

**两个条件同时满足**：
1. 多个线程**同时**对 HashMap 进行 put 操作
2. put 触发了扩容（resize / transfer）

### 5.2 推演过程（两线程并发扩容）

假设旧数组结构：`table[1] → A → B → null`

**单线程正常扩容**（头插法）：
```
遍历 A: newTable[1] = A, A.next = null        →  newTable[1]: A
遍历 B: newTable[1] = B, B.next = A           →  newTable[1]: B → A
```
结果：`B → A → null`（顺序反转，但正常）

**两线程并发时的灾难**：

```
时间线 ────────────────────────────────────────────────────────

线程1: e=A, next=B  →  挂起
线程2: 完整执行完迁移  →  newTable[1]: B → A → null

线程1 恢复: e=A, next=B
         → A.next = newTable[1] = B   →  A → B
         → newTable[1] = A            →  newTable[1]: A → B
         → e = B (next 仍是原来保存的 B)

         → B.next = newTable[1] = A   →  B → A
         → newTable[1] = B            →  B → A → B → A → …  (环形!)
```

**关键**：线程 1 的 `next` 指针是在线程 2 修改链表**之前**保存的，线程 2 改变了链表结构后，线程 1 按旧 next 指针操作 → 形成环。

### 5.3 JDK 1.8 怎么解决的

1. **尾插法**：不动链头，追加尾部，不反转顺序
2. **高低位分流**：`hash & oldCap` 直接分两路，不纠缠
3. 两个线程相互干扰最多产生**数据丢失**（一个线程的 put 被另一个覆盖），但不会再产生环形链表和死循环

## 七、终极背诵总结（面试秒答）

1. **1.7 vs 1.8 一句话**：7 数组链表头插法，8 引入红黑树改尾插，hash 简化扩容更高效
2. **死循环根因**：头插法 + 多线程并发 resize → 链表反转形成环
3. **树化条件**：链表 ≥ 8 且数组 ≥ 64 → 红黑树；缺任一 → 扩容
4. **退化条件**：树节点 ≤ 6 → 退化为链表；7 是缓冲避免频繁转换
5. **泊松分布**：链表长度达 8 概率 < 千万分之一，真出现是异常信号
6. **loadFactor=0.75**：时间和空间的折中点，泊松分布最优区间
7. **容量 2 的幂**：位运算取模 + 均匀分布 + 扩容时只需 hash & oldCap 判断去向
