# Hashtable vs Collections.synchronizedMap vs ConcurrentHashMap 核心对比

## 一、总览对比表
| 对比项 | Hashtable | Collections.synchronizedMap | ConcurrentHashMap (JDK 8+) |
|--------|-----------|------------------------------|-----------------------------|
| **锁机制** | 方法级 synchronized（全表锁） | 对象锁（mutex 互斥锁） | CAS + 桶头节点 synchronized |
| **锁粒度** | 整张表（超级粗） | 整张 Map 对象 | 单个哈希桶（细粒度） |
| **并发能力** | 极低（串行化） | 低（串行化） | 极高（真正并行） |
| **底层结构** | 数组 + 链表 | 包装 HashMap | 数组 + 链表 + 红黑树 |
| **null key/value** | 不允许 | 不允许 | 不允许 |
| **迭代器安全** | fail-fast（抛并发异常） | fail-fast（抛并发异常） | 弱一致性（不抛异常） |
| **性能** | 最差 | 差 | 最好 |
| **适用场景** | 极低并发、兼容老代码 | 简单同步、低并发 | 高并发生产环境（首选） |

---

## 二、详细原理

### 1. Hashtable
- 所有方法（put/get/remove/size）都加 `synchronized`
- **同一时刻只能有一个线程操作整个 Map**
- 多线程全部阻塞排队，完全串行执行
- 相当于给整张表加一把独占锁

### 2. Collections.synchronizedMap
- 静态方法生成的**包装类**
- 内部用一个 `mutex` 对象做同步
- 所有操作都包裹 `synchronized (mutex)`
- 本质 = **全表独占锁**
- 性能与 Hashtable 几乎一样

### 3. ConcurrentHashMap（JDK 8+）
- **无锁 + 细粒度锁** 混合
    - 数组初始化/空桶插入：**CAS 无锁**
    - 哈希冲突：只锁**当前桶的头节点**
- 不同桶可以**完全并行写入**
- 读操作**完全无锁**
- 并发能力 = 桶数量级别

---

## 三、核心区别（面试必背）

### 1. 锁粒度差距
- Hashtable / synchronizedMap：**锁整张表**
- ConcurrentHashMap：**锁单个桶**

### 2. 并发能力差距
- Hashtable / synchronizedMap：多线程 = 排队执行
- ConcurrentHashMap：多线程 = 真正并行执行

### 3. 迭代器安全
- Hashtable / synchronizedMap：快速失败，遍历期间修改抛异常
- ConcurrentHashMap：**弱一致性**，不抛异常，不保证读到最新数据

### 4. 高并发选择
**生产环境 100% 用 ConcurrentHashMap**
synchronizedMap / Hashtable 只适合极低并发场景

---

## 四、一句话总结
- **Hashtable**：古老全表锁，性能差，不推荐
- **synchronizedMap**：包装全表锁，简单但并发差
- **ConcurrentHashMap**：细粒度锁 + CAS，高并发首选，性能碾压前两者
