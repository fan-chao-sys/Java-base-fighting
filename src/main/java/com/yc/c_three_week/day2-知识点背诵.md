## 一、线程安全问题的本质

### 1.1 线程不安全的三要素条件

```
线程安全问题 = 多个线程 + 共享可变变量 + 无同步保护
```

**三个条件缺一不可**：
1. **多线程**并发执行（单线程永远没有安全问题）
2. 操作**共享变量**（同时读写同一个堆对象/静态变量）
3. 没有**同步机制**保护（无 synchronized / Lock / volatile / AtomicXD）

### 1.2 避免线程安全的五种方式

| 方式 | 原理 | 代表 |
|---|---|---|
| **无共享** | 每个线程有自己的变量副本，彻底不共享 | `ThreadLocal` / 栈封闭 |
| **不可变** | 变量一旦创建不能修改 | `final` / `String` / `Integer` |
| **互斥同步** | 同一时刻只有一个线程访问 | `synchronized` / `ReentrantLock` |
| **volatile** | 保证可见性（不保证原子性） | `volatile` 修饰共享标志位 |
| **原子类** | CAS 无锁保证单个变量的原子操作 | `AtomicInteger` / `LongAdder` |

## 二、并发编程三大特性（面试重中之重）

### 2.1 原子性（Atomicity）

**定义**：一个或多个操作，要么全部执行且不被中断，要么全部不执行。

**经典反例 — i++ 不是原子操作**：

```java
// i++ 在字节码层面是三步：
// 1. getfield    （读 i 的值）
// 2. iconst_1 / iadd （加 1）
// 3. putfield    （写回 i）
// 线程 A 刚完成步骤 2，线程 B 读到的是旧值 → 两个线程各加 1 次，最终结果只 +1
```

**解决方案**：
- `synchronized` / `ReentrantLock` → 加锁让三步串行
- `AtomicInteger.getAndIncrement()` → CAS 保证原子自增

### 2.2 可见性（Visibility）

**定义**：一个线程修改了共享变量后，其他线程**立即可见**。

**问题根源 — CPU 三级缓存 + JMM 工作内存**：

```
线程 A（CPU0）                       线程 B（CPU1）
  │                                     │
  │  i = 1 写入 CPU0 Cache              │
  │  未刷到主内存                        │ 读取 i 从 CPU1 Cache → 仍为 0
  │                                     │
  ▼                                     ▼
  每个 CPU 有独立 L1/L2 缓存，数据不一致
```

**解决方案**：
- `volatile` 修饰变量 → 写立即刷主内存，读从主内存读
- `synchronized` / `Lock` → 锁释放前刷主内存，获取锁后从主内存重新加载
- `final` → 构造完成对外可见，不需要同步

### 2.3 有序性（Ordering）

**定义**：程序执行的顺序，按照代码书写顺序执行。

**问题根源 — 指令重排序**：

```java
// 编译器 / JIT / CPU 三者都可能重排序

// 经典场景：DCL 单例中的重排序
instance = new Singleton();
// 这行代码在 CPU 层面分三步：
// ① 分配内存空间
// ② 调用构造器初始化对象
// ③ 将 instance 引用指向分配的内存地址
// ② 和 ③ 可能被 CPU 重排序 → 其他线程拿到"非 null 但未初始化"的对象！

// 对象半初始化的问题：
// 不加 volatile 时，线程在 ③ 执行后但 ② 未完成时访问 instance → 读到不完整的对象
```

**解决方案**：
- `volatile` → **禁止指令重排**（内存屏障）
- `synchronized` → 锁内代码串行化天然有序

### 2.4 三大特性总结

| | 问题 | 解决方案 |
|---|---|---|
| **原子性** | i++ 三步被打断 | `synchronized` / `Lock` / `AtomicXxx` |
| **可见性** | 一个线程改，另一个看不到 | `volatile` / `synchronized` / `Lock` |
| **有序性** | 指令重排，拿到半初始化对象 | `volatile` / `synchronized` |

## 三、JMM（Java 内存模型）底层吃透

### 3.1 JMM 核心架构

```
┌──────────────────────────────────────────────────────┐
│                    主内存（Main Memory）               │
│             所有线程共享，存实例字段、静态字段、数组元素   │
└──────┬───────────────────────────────────┬───────────┘
       │                                   │
  read │                        write ▲    │  read │                        write ▲
       ▼                              │    │       ▼                              │
┌──────────────┐            ┌──────────────┐
│ 工作内存（线程A）│            │ 工作内存（线程B）│
│ 私有变量副本    │            │ 私有变量副本    │
│ CPU缓存+寄存器  │            │ CPU缓存+寄存器  │
└──────────────┘            └──────────────┘
```

**核心规则**：
1. 线程操作变量必须先从主内存 **拷贝到工作内存**
2. 修改后必须**刷回主内存**，其他线程才能看到
3. 线程间的通信本质是**通过主内存做中转**

### 3.2 JMM 的 8 种原子操作（面试理解用）

| 操作 | 作用 | 方向 |
|---|---|---|
| lock | 锁定主内存变量，标记为线程独占 | 主内存 |
| unlock | 解锁 | 主内存 |
| read | 从主内存读取数据 | 主内存 → 工作内存 |
| load | 将 read 的值放入工作内存副本 | 工作内存 |
| use | 线程使用工作内存中的值 | 工作内存 → 执行引擎 |
| assign | 线程赋值到工作内存副本 | 执行引擎 → 工作内存 |
| store | 将工作内存值传递到主内存 | 工作内存 → 主内存 |
| write | 将 store 的值写入主内存变量 | 主内存 |

**记忆顺序**：`lock → read → load → use → assign → store → write → unlock`

### 3.3 happens-before 规则（8 条核心）

happens-before 是 JMM 定义的操作之间的偏序关系，保证前一个操作对后一个操作可见。

| 规则 | 内容 | 通俗理解 |
|---|---|---|
| **程序次序** | 单线程内，前面的操作 happens-before 后面的操作 | 单线程按代码顺序执行 |
| **锁规则** | 解锁 happens-before 后续加锁 | `unlock` 后的更改，下个 `lock` 可见 |
| **volatile** | volatile 写 happens-before 后续 volatile 读 | volatile 变量的修改立刻对读可见 |
| **线程启动** | `thread.start()` happens-before 线程内的任何操作 | 主线程的赋值，新线程能看到 |
| **线程终止** | 线程内所有操作 happens-before `thread.join()` 返回 | join 返回后能看到线程内的修改 |
| **中断规则** | `interrupt()` happens-before 检测到中断 | 被中断的线程能感知到标记 |
| **传递性** | A happens-before B，B happens-before C → A happens-before C | 链式可见 |
| **对象终结** | 构造函数执行完 happens-before `finalize()` 开始 | 构造完才可能被 GC 回收 |

## 四、指令重排序深度解析

### 4.1 三个层次的重排序

```
源码
  │  ① 编译器优化重排序（javac / JIT）
  ▼
字节码
  │  ② 指令级并行重排序（CPU 流水线 / 乱序执行）
  ▼
机器指令
  │  ③ 内存系统重排序（CPU 缓存 / Store Buffer）
  ▼
最终执行
```

### 4.2 经典重排场景

```java
// 场景1：普通变量赋值重排
int a = 1;   // ①
int b = 2;   // ②
// ① 和 ② 的顺序可能被重排（因为没有依赖关系）

// 场景2：DCL 单例中的致命重排
public class Singleton {
    private volatile static Singleton instance;  // ← volatile 是关键！
    // ...
    instance = new Singleton();
    // 三步: 分配内存 → 初始化 → 赋值给 instance
    // 无 volatile 时，初始化 和 赋值 可能重排
    // → 其他线程读到非 null 但未初始化的对象
}
```

## 五、高频易错点

### 1. volatile 能保证原子性吗？
**不能**。`i++` 不是原子操作，volatile 只保证每次读都是最新值，不保证多线程的"读-改-写"不被中断。用 `AtomicInteger`。

### 2. synchronized 能保证可见性吗？
**能**。JMM 规定：线程进入 synchronized 块前清空工作内存 → 从主内存重新加载 → 解锁前将修改刷新到主内存。

### 3. 所有共享变量都需要加 volatile 吗？
不是。只有在**一个线程写、多个线程读**的场景才需要。`synchronized` / `Lock` 已经保证了可见性，内部变量不需要 volatile。

## 六、终极背诵总结

1. **线程不安全三要素**：多线程 + 共享可变 + 无同步；缺一不可
2. **原子性**：操作不可分割（锁 / CAS解决）
3. **可见性**：一个线程修改别人看到（volatile / synchronized解决）
4. **有序性**：程序按代码顺序执行（volatile 内存屏障解决）
5. **JMM**：主内存共享，线程工作内存私有，8 种原子操作串联
6. **happens-before**：JMM 的可见性保证规则，8 条规则构成偏序链
7. **指令重排**：编译器→JIT→CPU 都参与，volatile 的 StoreLoad 屏障兜底
