## 一、JVM 运行时数据区 5 大块

### 1.1 整体架构图

```
┌─────────────────────────────────────────────────────────┐
│                    JVM 运行时数据区                        │
├───────────────┬───────────────────┬─────────────────────┤
│   线程私有     │     线程私有       │      线程共享         │
├───────────────┼───────────────────┼─────────────────────┤
│ 程序计数器     │    虚拟机栈        │       堆             │
│ (PC Register) │  (VM Stack)      │     (Heap)          │
│               │                  │  ┌────┬────┬────┐   │
│ 本地方法栈     │                  │  │Eden│ S0 │ S1 │   │
│ (Native Stack)│                  │  ├────┴────┴────┤   │
│               │                  │  │   老年代      │   │
│               │                  │  └─────────────┘   │
│               │                  │                     │
│               │                  │     方法区/元空间     │
│               │                  │   (Method Area /    │
│               │                  │    Metaspace)       │
└───────────────┴───────────────────┴─────────────────────┘
```

### 1.2 五大区域逐一吃透

| 区域 | 线程 | 存储内容 | 异常 | 核心要点 |
|---|---|---|---|---|
| **程序计数器** | 私有 | 当前线程执行的**字节码行号**指示器 | 唯一无 OOM 的区域 | 分支/循环/跳转/异常处理都依赖它 |
| **虚拟机栈** | 私有 | 一个个**栈帧**（Stack Frame） | StackOverflowError（深度超限）/ OOM | 每个方法调用 = 压入一个栈帧 |
| **本地方法栈** | 私有 | Native 方法的栈帧 | 同虚拟机栈 | HotSpot 把它和虚拟机栈合二为一 |
| **堆** | **共享** | 对象实例 + 数组 | **OOM: Java heap space** | GC 主战场，物理划分年轻代+老年代 |
| **方法区** | **共享** | 类信息、常量、静态变量、JIT 代码缓存 | OOM: Metaspace | JDK8+ 叫元空间，使用**本地内存** |

### 1.3 栈帧内部结构

```
每个栈帧 = ┌──────────────────┐
          │ 局部变量表         │ ← 基本类型值 + 对象引用（slot 为最小单位）
          │ Local Variables   │
          ├──────────────────┤
          │ 操作数栈           │ ← 字节码指令执行的计算空间（入栈/出栈）
          │ Operand Stack    │
          ├──────────────────┤
          │ 动态链接           │ ← 指向运行时常量池中该方法的符号引用
          │ Dynamic Link     │
          ├──────────────────┤
          │ 返回地址           │ ← 方法执行完后回到调用者的位置
          │ Return Address   │
          └──────────────────┘
```

## 二、堆的结构（分代模型）

### 2.1 物理划分

```
                    堆内存
┌────────────────────────────────────────────┐
│              年轻代（Young Gen）             │  默认占 1/3
│  ┌──────┬──────┬──────┐                    │
│  │ Eden │  S0  │  S1  │    默认 8 : 1 : 1  │
│  └──────┴──────┴──────┘                    │
├────────────────────────────────────────────┤
│              老年代（Old Gen）               │  默认占 2/3
│          存放长期存活的对象                    │
└────────────────────────────────────────────┘
```

### 2.2 分代比例公式

```
-XX:NewRatio=2      → 老年代:年轻代 = 2:1（年轻代占 1/3）
-XX:SurvivorRatio=8 → Eden:S0:S1 = 8:1:1
```

**为什么分代？** 对象"朝生夕灭"的假设 → 年轻代用复制算法效率最高 → 老年代存活率高用标记-整理。

## 三、对象创建流程（5 步完整链路）⭐⭐

```
new Object() 的全过程：

① 类加载检查
   检查该类的符号引用 → 常量池定位 → 是否已加载/解析/初始化
   未加载 → 先执行类加载

② 分配内存
   堆中划分一块空间给新对象
   ┌─ 指针碰撞（Bump the Pointer）：堆内存规整时，指针后移即可 → Serial/ParNew
   └─ 空闲列表（Free List）：内存不规整时，从空闲列表找 → CMS

   并发分配的安全性保证：
   ┌─ CAS + 失败重试
   └─ TLAB（Thread Local Allocation Buffer）：每个线程在 Eden 预设小空间

③ 初始化零值
   将分配到的内存空间全部初始化为零值（0 / 0.0 / false / null）
   保证对象的实例字段不赋初值也能使用

④ 设置对象头
   ┌─ Mark Word：hashCode / GC 分代年龄 / 锁状态标志 / 偏向线程ID 等
   └─ 类型指针（Klass Pointer）：指向方法区中该类的元数据

⑤ 执行 <init>
   调用构造方法，按代码赋值给实例字段
   → new 指令之后才执行 <init>（赋值顺序：父类构造→子类成员→子类构造）
```

**口诀**：加载 → 分配 → 清零 → 设头 → 初始化

## 四、对象内存布局（3 部分）

```
┌───────────────────┐
│    对象头          │
│  ┌─────────────┐  │
│  │  Mark Word  │  │  32/64位：hashCode、GC分代年龄、锁状态、偏向锁信息
│  ├─────────────┤  │
│  │Klass Pointer│  │  指向方法区 Class 元数据的指针（压缩后 32 位）
│  └─────────────┘  │
├───────────────────┤
│   实例数据          │  各字段值（含父类继承的字段）
│   Instance Data   │  JVM 自动排列：相同宽度的字段放在一起 → 减少填充
├───────────────────┤
│   对齐填充          │  保证对象起始地址是 8 字节的整数倍
│   Padding         │  HotSpot 自动内存管理要求
└───────────────────┘
```

**为什么需要对齐填充？** HotSpot 的内存管理要求对象起始地址必须是 8 字节的整数倍，便于 CPU 高效寻址。

## 五、JDK 8 方法区变迁

| | JDK 7 及之前 | JDK 8+ |
|---|---|---|
| **名称** | 永久代（PermGen） | **元空间（Metaspace）** |
| **存储位置** | JVM 堆内 | **本地内存（Native Memory）** |
| **大小** | `-XX:PermSize` / `-XX:MaxPermSize`（默认小，容易 OOM） | `-XX:MetaspaceSize` / `-XX:MaxMetaspaceSize`（默认无上限） |
| **字符串常量池** | 在永久代 | **移到堆** |
| **好处** | — | ① 不再受 JVM 堆大小限制；② 避免 PermGen OOM |

**为什么移出**：永久代大小固定，大量动态生成类（如 JSP/CGLIB）容易 PermGen OOM。元空间用本地内存，由 OS 管理。

## 六、JVM 内存结构 vs JMM（面试高频辨析）

| | JVM 内存结构 | JMM（Java 内存模型） |
|---|---|---|
| **本质** | **物理/实际**的内存区域划分 | **逻辑/抽象**的并发访问规范 |
| **关注点** | "数据放哪" — 堆/栈/方法区 | "多线程怎么协调访问" — 可见性/有序性/原子性 |
| **组成部分** | 堆、栈、方法区、程序计数器、本地方法栈 | 主内存 + 工作内存 + 8 种原子操作 + happens-before |
| **作用** | 内存分配和 GC 的基础 | 并发编程正确性的保证 |

> **一句话区别**：JVM 内存结构是实体的分区，JMM 是并发访问的规则。

## 七、终极背诵总结

1. **5 大区域**：堆（共享/对象）、方法区（共享/类元数据）、栈（私有/栈帧）、本地方法栈、程序计数器
2. **堆分代**：年轻代 Eden:S0:S1=8:1:1 + 老年代，分代是为了用最合适的 GC 算法
3. **对象创建 5 步**：类加载检查 → 分配内存（指针碰撞/空闲列表/TLAB）→ 初始化零值 → 设置对象头 → 执行 `<init>`
4. **对象布局 3 段**：对象头（Mark Word + Klass Pointer）→ 实例数据 → 对齐填充
5. **JDK 8 元空间**：永久代 → 元空间（本地内存），字符串常量池移到堆
6. **JVM 内存 vs JMM**：前者是物理划分（数据放哪），后者是逻辑规范（线程怎么访问）
