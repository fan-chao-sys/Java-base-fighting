## 一、如何判断对象可回收

### 1.1 两种判定算法

| 算法 | 原理 | 优点 | 缺点 |
|---|---|---|---|
| **引用计数法** | 每个对象维护引用计数器 +1/-1 → 为 0 回收 | 实现简单、判定快 | **循环引用**永远不为 0 → 内存泄漏 |
| **可达性分析**（Java 采用） | 从 **GC Roots** 出发，沿引用链遍历，不可达的对象判定为可回收 | 解决循环引用 | 需要 STW 保证快照一致性 |

### 1.2 循环引用问题

```java
class A { B b; }
class B { A a; }
A a = new A();
B b = new B();
a.b = b;
b.a = a;  // 循环引用
a = null;
b = null;
// 引用计数法：a 和 b 的计数各为 1（互相引用），永远不回收！
// 可达性分析：a 和 b 从 GC Roots 都不可达 → 正确回收
```

### 1.3 GC Roots 包括哪些（5 类）⭐⭐

```
GC Roots 对象：
  ① 虚拟机栈（栈帧的局部变量表）中引用的对象 → 当前正在执行的方法中的变量
  ② 方法区中 static 变量引用的对象 → 类的静态字段
  ③ 方法区中常量池引用的对象 → String 常量池中的引用
  ④ 本地方法栈中 JNI 引用的对象 → Native 代码中的全局引用
  ⑤ synchronized 持有的对象 → 被锁定的对象
```

**口诀**：栈中的引用、静态变量、常量池、JNI 引用、锁持有的对象

## 二、四种引用类型（必背）⭐⭐⭐

| 引用类型 | 回收时机 | GC 行为 | 典型场景 |
|---|---|---|---|
| **强引用 Strong** | **永不回收** | 宁愿 OOM 也不回收 | 普通 `new` 出来的对象 |
| **软引用 Soft** | **OOM 前**第二次 GC 时回收 | 内存够用就留着；不够就回收 | 图片缓存、网页缓存 |
| **弱引用 Weak** | **下次 GC 必回收** | 只要 GC 就回收 | `WeakHashMap`、`ThreadLocal` 的 key |
| **虚引用 Phantom** | 随时 | 无法通过它获取对象实例，只配合 `ReferenceQueue` 收通知 | 堆外内存管理（`DirectByteBuffer`） |

### 2.1 代码示例

```java
// 强引用
Object obj = new Object();  // obj 强引用 → 永远不会被 GC

// 软引用
SoftReference<byte[]> soft = new SoftReference<>(new byte[10 * 1024 * 1024]);
byte[] data = soft.get();   // OOM 之前有值，OOM 之后返回 null

// 弱引用
WeakReference<Object> weak = new WeakReference<>(new Object());
System.gc();
Object o = weak.get();      // → null（GC 后立即被回收）

// 虚引用 —— 必须配合 ReferenceQueue
ReferenceQueue<Object> queue = new ReferenceQueue<>();
PhantomReference<Object> phantom = new PhantomReference<>(new Object(), queue);
// phantom.get() 永远返回 null → 虚引用不能获取对象实例
// 对象被回收时，phantom 被放入 queue → 可以知道对象何时被回收
```

### 2.2 引用的强度链

```
强引用 > 软引用 > 弱引用 > 虚引用
  │        │        │        │
  不死      快死时死  一 GC 就死  已经死了（只是收尸通知）
```

## 三、三种垃圾回收算法

### 3.1 标记-清除（Mark-Sweep）

```
标记阶段：从 GC Roots 遍历，标记可达对象
清除阶段：遍历堆，回收未标记对象

┌──┬──┬──┬──┬──┬──┐  标记后：  ┌──┬  ┬──┬  ┬──┬──┐
│A │B │C │D │E │F │          │A │  │C │  │E │F │
└──┴──┴──┴──┴──┴──┘          └──┴──┴──┴──┴──┴──┘
       回收 B、D                        碎片化!
```

**优点**：实现简单，不需要移动对象。
**缺点**：**内存碎片**严重，分配大对象时找不到连续空间 → 提前触发下一次 GC。

### 3.2 标记-整理（Mark-Compact）

```
标记阶段：同标记-清除，标记可达对象
整理阶段：将所有存活对象向一端移动，直接清掉边界外的内存

┌──┬  ┬──┬  ┬──┬──┐  整理后：  ┌──┬──┬──┬──┐
│A │  │C │  │E │F │          │A │C │E │F │  连续空间
└──┴──┴──┴──┴──┴──┘          └──┴──┴──┴──┘  ↑ 无碎片
         碎片化                              可用指针碰撞分配
```

**优点**：消除碎片，内存连续。
**缺点**：移动对象需要 STW + 更新所有引用（CPU 成本高）。

### 3.3 复制算法（Copying）

```
┌────────┬────────┐        ┌────────┬────────┐
│  From  │   To   │   →   │   To   │  From  │
│  使用中 │  空闲   │       │  使用中 │  空闲   │
│  A C E │        │       │  A C   │        │
└────────┴────────┘       └────────┴────────┘
  回收 B、D  →  把 A、C 复制到 To → 清空 From → From/To 互换
```

**优点**：无碎片，分配用指针碰撞，极快。
**缺点**：**浪费一半空间**（始终有一半空着）。

### 3.4 三种算法总结

| 算法 | 碎片 | 空间利用率 | 适用区域 | 为什么 |
|---|---|---|---|---|
| **标记-清除** | 有 | 高 | 老年代（CMS） | 存活对象多，不浪费空间 |
| **标记-整理** | 无 | 高 | 老年代（Serial Old） | 无碎片，但需移动 |
| **复制算法** | 无 | 低（50%） | **年轻代** | 对象朝生夕灭，只需复制少量存活对象 |

## 四、分代回收思想

### 4.1 为什么分代？

基于两个**弱分代假说**：
1. **绝大多数对象朝生夕灭**（Weak Generational Hypothesis）→ 年轻代用复制算法，效率极高
2. **熬过多次 GC 的对象更难消亡**（Strong Generational Hypothesis）→ 老年代用标记-清除/整理

### 4.2 分代回收流程

```
1. new 对象 → Eden 区
2. Eden 满 → Minor GC
   ├─ 存活对象 → 复制到 Survivor（S0），年龄 = 1
   └─ Eden 清空
3. 下一次 Eden 满 → Minor GC
   ├─ Eden 存活对象 + S0 存活对象 → 复制到 S1，年龄各自 +1
   └─ Eden + S0 清空，S0/S1 角色互换
4. 每次 Minor GC，存活对象在 S0/S1 之间拷贝，年龄累加
5. 年龄 ≥ MaxTenuringThreshold(15) → 晋升到老年代
6. 老年代满 → Major GC / Full GC
```

### 4.3 Minor GC vs Major GC vs Full GC

| | 触发区域 | 触发条件 | STW 时间 |
|---|---|---|---|
| **Minor GC** | 年轻代 | Eden 满 | 短（复制少量存活对象） |
| **Major GC** | 老年代 | 老年代满 | 长（遍历所有老年代对象） |
| **Full GC** | 整个堆 + 方法区 | 老年代/元空间不足、System.gc() | 最长（STW 全局清理） |

## 五、内存分配与回收策略速记

```
对象 → Eden（优先）→ Survivor（Minor GC 后存活）→ Old（长期存活/大对象）

回收：
Eden + S0 → (Minor GC) → 存活 → S1（S0/S1 轮换）
                         → 年龄够/放不下 → Old
Old → (Major GC / Full GC) → 清理无引用对象
```

## 六、终极背诵总结

1. **可达性分析**：从 GC Roots 出发，不可达即回收；解决循环引用
2. **GC Roots 5 类**：栈引用、static 变量、常量池、JNI 引用、锁持有对象
3. **4 种引用**：强（不回收）→ 软（OOM 前回收）→ 弱（GC 必回收）→ 虚（仅跟踪回收通知）
4. **3 种算法**：年轻代复制（无碎片/浪费一半）、老年代标记-清除（有碎片）/标记-整理（无碎片/需移动）
5. **分代思想**：对象朝生夕灭 → 年轻代复制效率高；熬过多次 GC → 老年代存量大 → 标记-整理
6. **GC 分类**：Minor GC（年轻代，快）、Major GC（老年代，慢）、Full GC（全局，最慢）
