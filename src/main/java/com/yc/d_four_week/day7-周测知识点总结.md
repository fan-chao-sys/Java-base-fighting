## 一、10 道自测题（口述标准答案）

### Q1: JVM 运行时内存区域包含哪些？各干什么？

**答：**

| 区域 | 线程 | 存储内容 | 异常 |
|---|---|---|---|
| **堆（Heap）** | 共享 | 对象实例、数组 | OOM: Java heap space |
| **方法区 / 元空间** | 共享 | 类信息、常量、静态变量、JIT 缓存 | OOM: Metaspace |
| **虚拟机栈（VM Stack）** | 私有 | 栈帧（局部变量表、操作数栈、动态链接、返回地址） | StackOverflowError |
| **本地方法栈** | 私有 | Native 方法的栈帧 | StackOverflowError |
| **程序计数器（PC Register）** | 私有 | 当前执行的字节码行号 | 唯一无 OOM 的区域 |

**堆分代结构**：年轻代 Eden:S0:S1 = 8:1:1 + 老年代。JDK 8 方法区从永久代迁移到元空间（本地内存）。

**本周验证（W4D1_Actual）**：使用 jol-core 打印对象内存布局，观察对象头（Mark Word + Klass Pointer）+ 实例数据 + 对齐填充。

### Q2: JVM 内存结构和 JMM 有什么区别？

**答：**

| | JVM 内存结构 | JMM（Java 内存模型） |
|---|---|---|
| **本质** | **物理/实际**的内存区域划分 | **逻辑/抽象**的并发访问规范 |
| **关注点** | "数据放哪" — 堆/栈/方法区 | "多线程怎么访问" — 可见性/有序性/原子性 |
| **组成** | 堆、栈、方法区、程序计数器、本地方法栈 | 主内存 + 工作内存 + 8种原子操作 + happens-before |
| **作用** | 内存分配和 GC 的基础 | 并发编程正确性的保证 |

**一句话**：JVM 内存结构是实体的分区，数据存在哪；JMM 是并发的规则，多线程怎么访问共享数据。

### Q3: 类加载 5 个阶段分别做什么？

**答：**

```
加载 → 验证 → 准备 → 解析 → 初始化
```

| 阶段 | 核心动作 | 关键点 |
|---|---|---|
| **加载** | 通过全限定名获取字节流 → 方法区生成运行时结构 → 堆中创建 `Class` 对象 | 来源：class文件/jar/网络/动态生成 |
| **验证** | 文件格式（CAFEBABE）+ 元数据 + 字节码 + 符号引用验证 | JVM 安全防线，拒绝恶意字节码 |
| **准备** | 为 static 变量分配内存并赋**零值** | **例外**：`static final` 常量直接赋代码值 |
| **解析** | 符号引用 → 直接引用（内存地址/偏移量） | 类/字段/方法/接口 4 类符号引用 |
| **初始化** | 执行 `<clinit>()`（static 赋值 + static 代码块，按顺序） | 父类先于子类，多线程同步保证只执行一次 |

**本周验证（W4D3_Actual）**：打印 AppClassLoader → PlatformClassLoader → BootstrapClassLoader 三层类加载器的加载路径，并用 `MyCustomClassLoader` 自定义类加载器从指定目录加载 class 文件。

### Q4: 双亲委派模型的好处？什么场景打破它？

**答：**

**三层架构**：
```
AppClassLoader（应用） → PlatformClassLoader（平台） → BootstrapClassLoader（启动）
        自底向上查缓存                            自顶向下试加载
```

**好处**：
1. **防止核心类被篡改**：自定义 `java.lang.String` 不会生效，Bootstrap 先加载到
2. **避免重复加载**：全限定名 + 类加载器确定唯一性
3. **安全隔离**：核心 API 由信任的加载器加载

**打破双亲委派的场景**：

| 场景 | 为什么打破 | 怎么做的 |
|---|---|---|
| **SPI（如 JDBC）** | 接口在 rt.jar（Bootstrap加载），实现在用户 classpath | 线程上下文类加载器 `Thread.setContextClassLoader()` |
| **Tomcat** | 多个 Web 应用需要类隔离（各自不同版本的 jar） | `WebAppClassLoader` 打破委派，先自己加载 |
| **OSGi** | 模块化，每个 Bundle 独立类加载器 | 网状委派 |
| **热部署** | 同一个类重新加载不重启 JVM | 自定义 ClassLoader 每次读新 class |

### Q5: 对象什么时候进入老年代（4 种情况）？

**答：**

| 序号 | 触发条件 | 机制 |
|---|---|---|
| **① 大对象** | 对象大小 > `-XX:PretenureSizeThreshold` | 直接老年代分配，避免在 Eden/Survivor 间复制 |
| **② 年龄达标** | 熬过 Minor GC 次数 ≥ `MaxTenuringThreshold`（默认 15） | 每活过一次 Minor GC 年龄+1 |
| **③ 动态年龄判断** | Survivor 中**同龄对象总大小 > Survivor 的 50%** | 该年龄及以上全部晋升，不等到 15 |
| **④ 分配担保失败** | Minor GC 后 Survivor 放不下存活对象 | 多余对象直接进老年代 |

### Q6: 标记-清除、标记-整理、复制算法各用在什么区域？

**答：**

| 算法 | 适用区域 | 原因 | 优点 | 缺点 |
|---|---|---|---|---|
| **复制算法** | **年轻代** | 对象朝生夕灭（98% 死亡），只需复制少量存活对象 | 无碎片，分配用指针碰撞极快 | 浪费一半空间 |
| **标记-清除** | 老年代（CMS） | 存活对象多，不浪费空间 | 不移动对象，效率高 | **碎片化** |
| **标记-整理** | 老年代（Serial Old / Parallel Old） | 无碎片，但需移动 | 内存连续，无碎片 | STW 时间长（需更新引用） |

**为什么年轻代不用标记-整理？** 年轻代存活率极低，复制少量存活对象比全局扫描+移动所有存活对象效率高得多。

### Q7: G1 的 Region 设计有什么优势？

**答：**

**Region 核心思想**：把堆划分为约 2048 个等大 Region（1~32MB），不再物理隔离年轻代/老年代。

```
传统分代：                      G1 Region：
┌──────────────┐              ┌──┬──┬──┬──┬──┬──┐
│    老年代     │              │E │S │O │E │H │O │  每个 Region 一种角色
├──────────────┤              ├──┼──┼──┼──┼──┼──┤  E=Eden, S=Survivor
│S0│S1│ Eden  │              │O │E │O │E │O │E │  O=Old, H=大对象
└──────────────┘              └──┴──┴──┴──┴──┴──┘
```

**5 大优势**：
1. **不再物理隔离**：年轻代/老年代 Region 可动态转换，灵活调整比例
2. **Mixed GC**：只回收**部分**垃圾最多的老年代 Region，不是全堆扫描
3. **可预测停顿**：`-XX:MaxGCPauseMillis` 控制每次回收的 Region 数量
4. **无碎片**：使用复制/整理算法，回收时把存活对象搬到空 Region
5. **增量回收**：每次只回收在预估时间内的 Region，避免一次性全堆 STW

### Q8: CMS 和 G1 的核心区别？

**答：**

| 对比维度 | CMS | G1 |
|---|---|---|
| **算法** | 老年代：标记-清除 | 整体：标记-整理 + 复制（Region 间） |
| **内存布局** | 物理分年轻代/老年代 | **Region 化**，不分物理边界 |
| **碎片化** | **有**（标记-清除不整理） | **无**（复制/整理） |
| **可预测停顿** | 不可以 | **可以**（MaxGCPauseMillis=200ms） |
| **回收粒度** | 全老年代（Full GC 时） | 部分 Region（垃圾最多的优先） |
| **适用堆大小** | ≤ 4-8G | **≥ 6G** |
| **默认** | JDK 14 已移除 | **JDK 9+ 默认** |
| **并发失败** | Concurrent Mode Failure → Serial Old | Evacuation Failure → 退化 Full GC |

**CMS 四阶段**：初始标记(STW短) → 并发标记(不STW) → 重新标记(STW中) → 并发清除(不STW)

**G1 回收阶段**：Young GC → **Mixed GC**（回收全年轻代 + 部分老年代垃圾最多的 Region）

### Q9: Full GC 频繁时你的排查步骤？

**答：**

```
第 1 步：确认现状
  jps -l                        ← 找 PID
  jstat -gcutil <pid> 1000      ← 每秒看 FGC 次数和堆占比
  top -Hp <pid>                 ← 看线程 CPU

第 2 步：定位原因
  老年代增长快 → 检查晋升速率 / 大对象
  元空间不足 → 检查动态代理/CGLIB
  内存泄漏 → dump 分析

第 3 步：生成 Dump（重点）
  jmap -dump:format=b,file=heap.hprof <pid>
  → MAT 打开 → Leak Suspects → Dominator Tree → GC Roots 追溯

第 4 步：线程排查
  jstack <pid> > thread.txt
  → 找 BLOCKED / RUNNABLE / 死锁

第 5 步：止血 + 根治
  止血：重启 / 扩堆(-Xmx) / 限流
  根治：修泄漏代码 / 优化大对象 / GC 参数调优
```

**必备命令**：`jps` / `jstat -gcutil` / `jmap -dump` / `jstack` / `jinfo` / `jcmd`

**本周验证（W4D6_Actual）**：构造 OOM（无限 `list.add(new byte[1024*1024])`）+ `-XX:+HeapDumpOnOutOfMemoryError` 自动 dump + MAT 分析思路；死锁代码用 `jstack` 定位。

### Q10: 强引用/软引用/弱引用/虚引用的区别和使用场景？

**答：**

| 引用类型 | 回收时机 | GC 行为 | 使用场景 |
|---|---|---|---|
| **强引用 Strong** | **永不回收** | 宁愿 OOM 也不回收 | 普通 `new` 对象 |
| **软引用 Soft** | **OOM 前**回收 | 内存不够时第二次 GC 回收 | 图片/网页缓存 |
| **弱引用 Weak** | **下次 GC 必回收** | 无论内存够不够都回收 | `WeakHashMap`、`ThreadLocal` key |
| **虚引用 Phantom** | 随时 | `get()` 永远返回 null，仅配合 `ReferenceQueue` 收通知 | 堆外内存管理（`DirectByteBuffer`） |

**引用强度链**：强 > 软 > 弱 > 虚

**本周验证（W4D4_Actual）**：
- 强引用：持续 `new byte[1MB]` + `ArrayList` 持有 → OOM
- 软引用：包裹为 `SoftReference` + 置空强引用 → 内存不足时自动回收，不 OOM

### 2.5 GC 算法三选

| 年轻代 | 老年代 |
|---|---|
| **复制**（对象朝生夕灭，复制少量存活对象） | **标记-清除**（CMS，有碎片）/ **标记-整理**（Serial Old，无碎片） |

## 四、JVM 面试题口述稿

### 4.1 GC Roots 有哪些？

**5 类 GC Roots**：虚拟机栈引用、static 变量、常量池引用、JNI 引用、synchronized 持有的锁对象。

**口诀**：栈中的引用、类的静态字段、常量池、JNI、锁对象。

### 4.2 Minor GC / Major GC / Full GC 区别

| | Minor GC | Major GC | Full GC |
|---|---|---|---|
| **区域** | 年轻代 | 老年代 | 整个堆 + 方法区 |
| **触发** | Eden 满 | 老年代满 | 多种原因 |
| **速度** | 快 | 慢 | 最慢 |
| **算法** | 复制 | 标记-清除/整理 | 取决于 GC 器 |

### 4.3 对象分配担保机制

Minor GC 前，JVM 先检查**老年代最大可用连续空间 > 年轻代所有对象总和**（或历次晋升平均大小），不满足则直接触发 Full GC 给老年代腾空间。

### 4.4 内存泄漏 vs OOM

- **OOM**：内存确实不够用了（对象太多或堆太小）
- **内存泄漏**：垃圾对象没有被回收（有引用一直指着），可用内存越来越少
- 内存泄漏 → 堆慢慢被占满 → 频繁 Full GC → 最终 OOM

### 4.5 jstat -gcutil 输出字段速记

```
S0  S1  E   O   M   CCS  YGC  YGCT  FGC  FGCT  GCT
│   │   │   │   │   │    │    │     │    │     │
Surv Surv Eden Old Meta 压  YGC 耗时  FGC 耗时  总耗时
  0    1             缩
```

**生产铁律**：FGC 列必须为 0，出现 Full GC 必须排查。

### 4.6 逃逸分析的三大优化

| 优化 | 原理 | 效果 |
|---|---|---|
| **栈上分配** | 未逃逸对象分配在栈帧 | GC 压力为零 |
| **锁消除** | 对象单线程使用 → 去掉同步 | 消除不必要锁开销 |
| **标量替换** | 对象拆为基本字段，分配在栈/寄存器 | 连 new 都不用了 |

**本周验证（W4D2_Actual）**：循环 1 亿次创建未逃逸 User 对象，开启逃逸分析后几乎无 GC → 验证栈上分配效果。

## 六、速记口诀

1. **5 大区域**：堆管对象栈管帧，方法区存类元数据，程序计数指行号，本地方法管 Native
2. **对象创建**：加载检查 → 分配内存 → 初始零值 → 设置对象头 → 执行 init
3. **双亲委派**：自底向上查缓存，自顶向下试加载；防篡改、防重复、保安全
4. **进老年代 4 路**：大对象直进、年龄到 15、动态同龄过半、Survivor 放不下
5. **GC 算法分配**：年轻代复制（朝生夕灭）、老年代标清（CMS 有碎片）/ 标整（无碎片）
6. **G1 Region**：不分物理代、MixedGC 回收部分老年代、可预测停顿、无碎片增量回收
7. **CMS 四阶段**：初始标记短 STW → 并发标记长不 STW → 重新标记中 STW → 并发清除不 STW
8. **引用四级**：强不死、软 OOM 前回收、弱 GC 必回收、虚只收通知
9. **Full GC 排查**：jstat 看频率 → jmap dump → MAT 追根因 → 修代码调参数
