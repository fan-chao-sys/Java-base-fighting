## 一、B+ 树核心特性（为什么 MySQL 选它）

### 1.1 B+ 树结构

```
                ┌────────┐
                │ 30  60 │  ← 非叶子节点：只存索引键（不存数据行）
                └──┬──┬──┘
       ┌──────────┘  │  └──────────┐
       ▼              ▼              ▼
   ┌─10─20─┐    ┌─40─50─┐     ┌─70─80─90─┐   ← 叶子节点：存完整数据行
   │ data  │    │ data  │     │  data    │      且形成双向链表
   └───┬───┘    └───┬───┘     └────┬─────┘
       └────────────┴──────────────┘
            ← 双向链表，支持范围查询顺序遍历 →
```

### 1.2 B+ 树三大优势

| 特性 | 带来的好处 |
|---|---|
| **非叶子节点只存键** | 每页能存更多索引项 → **扇出更大 → 树更矮 → 磁盘 IO 更少** |
| **叶子节点存全部数据** | 每次查询都落到叶子节点，**查询时间稳定** O(log n) |
| **叶子节点双向链表** | **范围查询** (BETWEEN / > / <) 只需定位起点，沿链表顺序读取即可 |

### 1.3 B+ 树 vs B 树 vs 红黑树

| | B+ Tree | B Tree | 红黑树 |
|---|---|---|---|
| **非叶子存数据？** | ❌ 只存键 | ✅ 存键+数据 | ✅ 存键+数据（每个节点一个） |
| **叶子有链表？** | ✅ 双向链表 | ❌ | ❌ |
| **树高** | **矮**（高扇出） | 中等 | **高**（二叉树） |
| **磁盘 IO 次数** | **少** | 中等 | **多**（不适合磁盘） |
| **范围查询** | **极快**（链表遍历） | 需中序遍历回退 | 需中序遍历 |

**一句话**：红黑树高→磁盘 IO 多→不适合；B 树扇出小→范围查询无链表→不够好；**B+ 树扇出大+链表=磁盘最优**。

## 二、聚簇索引（Clustered Index）

### 2.1 核心定义

**数据即为索引，索引即为数据。** 叶子节点存储的是**完整的行数据**。一张表只能有一个聚簇索引（因为数据只能按一种顺序物理存储）。

### 2.2 InnoDB 聚簇索引的生成规则

```
优先级：
  ① 有 PRIMARY KEY → 用主键作为聚簇索引
  ② 无主键，有唯一非空索引 → 用第一个 UNIQUE NOT NULL 索引
  ③ 都没有 → InnoDB 自动生成隐藏列 row_id（6字节），自增
```

**结论**：InnoDB 表**一定有**聚簇索引，建议**显式建自增主键**（避免隐式 row_id 的额外开销和无法查询的问题）。

### 2.3 为什么推荐自增主键？

- **自增插入**：新行始终追加到 B+ 树最右边 → 不会引起页分裂 → 性能最好
- **随机主键（UUID）**：插入位置随机 → 频繁页分裂 → 页空间利用率低 → 性能差

## 三、二级索引（Secondary Index）

### 3.1 核心定义

**叶子节点只存索引列 + 主键值**（不是完整行数据）。

```
二级索引 B+ 树结构（以 idx_name 为例）：

 非叶子：存放 name 列的键值
 叶子  ：[name值, 主键id]       ← 只有 name + id，没有其他列的数据
```

拿到主键后 → 必须回到聚簇索引才能获取整行数据 → 这就是**回表**。

### 3.2 二级索引的查找路径

```
SELECT * FROM user WHERE name = '张三';

① 从 idx_name 二级索引查 '张三' → 得到主键 id=101
② 从聚簇索引查 id=101 → 得到完整行数据 (id,name,age,email,...)
                              ↑
                           这就是"回表"
```

---

## 四、回表（一次查询两次 B+ 树查找）

### 4.1 回表的代价

- 二级索引查找 → 1 次（可能又涉及多次页 IO）
- 聚簇索引查找 → 1 次
- **每次回表都是随机 IO** → 当需要回表的行数很多时，性能急剧下降
- 优化器预估回表行数 > 全表的 ~20% → 可能**放弃索引，改全表扫描**

### 4.2 如何避免回表？

**覆盖索引**：查询的所有列都在二级索引中 → 不需要回表。

```sql
-- 建联合索引 idx_name_age(name, age)

-- ❌ 回表查询
SELECT * FROM user WHERE name = '张三';
-- 需要 email 列 → 二级索引没有 → 必须回表

-- ✅ 覆盖索引查询
SELECT name, age FROM user WHERE name = '张三';
-- name 和 age 都在索引中 → 不用回表！
-- EXPLAIN Extra: Using index（覆盖索引标志）
```

### 4.3 EXPLAIN 中覆盖索引的标识

```
Extra: Using index  ← 表示"覆盖索引"，直接从二级索引返回结果，无需回表
Extra: Using index condition ← 使用了索引下推（ICP），部分过滤在引擎层完成
```

## 五、索引下推（ICP — Index Condition Pushdown）

### 5.1 什么是 ICP

MySQL 5.6+ 引入，把**能在索引中判断的 WHERE 条件，在引擎层直接过滤**，减少回表次数。

```sql
-- 联合索引 idx(name, age)
SELECT * FROM user WHERE name LIKE '张%' AND age = 25;

-- 没有 ICP：
--   先用 name LIKE '张%' 找到所有匹配记录 → 全部回表 → 在 Server 层过滤 age=25

-- 有 ICP：
--   先用 name LIKE '张%' 找到所有匹配记录
--   → 在引擎层直接用 age=25 过滤 → 只回满足两条的记录
--   → 大幅减少回表次数！
```

## 六、核心概念串联图

```
SELECT * FROM user WHERE name = '张三';

  ┌─────────────────┐
  │  idx_name (二级)  │ ← Extra = Using index condition (ICP 过滤)
  │  找到 '张三'+id  │
  └────────┬────────┘
           │
           ▼  拿到主键 id
  ┌─────────────────┐
  │  聚簇索引 (主键)  │ ← 这里就是"回表"：再去主键索引查一次
  │  拿到完整行数据    │
  └────────┬────────┘
           │
           ▼  返回数据
  ┌─────────────────┐
  │  Server 层       │ ← 返回客户端
  └─────────────────┘

  如果用覆盖索引（SELECT name FROM user WHERE name = '张三'）：
  直接在 idx_name 就拿到 name → 不用回表 → Extra: Using index
```

## 七、终极背诵总结

1. **B+ 树三优势**：非叶只存键（矮）→ 叶子存数据（稳）→ 双向链表（范围快）
2. **聚簇索引**：数据即索引，叶子存完整行，InnoDB 必须有一个
3. **二级索引**：叶子存索引键+主键，查完还得"回表"
4. **回表**：二级索引 → 拿到主键 → 聚簇索引 → 完整数据，额外一次 IO
5. **覆盖索引**：查询列全在索引里 → Extra: Using index → 不回表
6. **ICP**：索引层面先过滤 → 减少回表次数
7. **自增主键**：顺序追加，避免页分裂，性能最好
