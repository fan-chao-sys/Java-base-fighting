## 一、最左前缀规则（最核心）

### 1.1 底层原理

联合索引 (a, b, c) 的 B+ 树按 **a → b → c** 的顺序排序：

```
先按 a 排序 → a 相同的按 b 排序 → a、b 相同按 c 排序

(a=1, b=2, c=3)
(a=1, b=2, c=5)   ← b 相同，再排 c
(a=1, b=4, c=1)   ← a 相同，再排 b
(a=2, b=1, c=0)   ← a 不同，换下一个 a 区间
```

**物理结构决定**：a 的值分散在整个 B+ 树上，b 只在 a 相等的局部有序。没有 a，b 和 c 在整个树上乱序 → 无法用索引定位。

### 1.2 联合索引能匹配的查询

```sql
INDEX idx_abc(a, b, c)

✅ WHERE a = 1                     -- 用 (a)
✅ WHERE a = 1 AND b = 2           -- 用 (a, b)
✅ WHERE a = 1 AND b = 2 AND c = 3 -- 用 (a, b, c) 全部
✅ WHERE a = 1 AND c = 3           -- 用 (a)，c 断掉了
✅ WHERE a = 1 AND b > 2 AND c = 3 -- 用 (a, b)，c 断在范围查询

❌ WHERE b = 2                     -- 没有 a，全表扫描
❌ WHERE c = 3                     -- 没有 a，全表扫描
❌ WHERE b = 2 AND c = 3           -- 全表扫描
❌ WHERE a > 1 AND b = 2           -- 用 (a)，b 断在范围查询
```

**口诀**：**全值匹配我最爱，最左前缀要遵守；带头大哥不能死，中间兄弟不能断。**

## 二、索引失效 10 大场景（必背）⭐⭐⭐

| 序号 | 场景 | 示例 | 原因 |
|---|---|---|---|
| ① | **LIKE 以 % 开头** | `WHERE name LIKE '%张'` | B+ 树按前缀排序，`%` 开头无法定位起点 |
| ② | **隐式类型转换** | `WHERE phone = 13800138000`（phone 是 VARCHAR） | MySQL 把字段转为数字，索引失效 |
| ③ | **索引列上使用函数** | `WHERE DATE(create_time) = '2024-01-01'` | 索引是原始值，函数后变了 |
| ④ | **索引列上运算** | `WHERE id + 1 = 10` | 等价于 `WHERE id = 9`，但写错就失效 |
| ⑤ | **OR 中有非索引列** | `WHERE name = '张' OR age = 25`（age 无索引） | 一个条件不走索引，整体不走 |
| ⑥ | **不等于** `!=` / `<>` | `WHERE status != 1` | 需扫描大部分数据，优化器可能放弃索引 |
| ⑦ | **IS NULL / IS NOT NULL** | `WHERE name IS NULL` | 同不等于，扫描范围大 |
| ⑧ | **不满足最左前缀** | `WHERE b = 2 AND c = 3`（索引是 (a,b,c)） | B+ 树的物理排序决定 |
| ⑨ | **优化器选择全表扫描** | 回表行数 > 全表 ~20% | 全表比"回表+随机 IO"更快 |
| ⑩ | **ORDER BY 不满足最左前缀** | `ORDER BY b, c`（只有 (a,b,c) 索引） | 排序字段不在最左前缀的连续部分 |

### 2.1 LIKE 需要右侧模糊

```sql
-- ❌ 索引失效
WHERE name LIKE '%张'

-- ✅ 走索引
WHERE name LIKE '张%'

-- ❌ 也失效
WHERE name LIKE '%张%'
```

### 2.2 类型转换的陷阱

```sql
-- phone 列是 VARCHAR(11)
-- ❌ 索引失效（MySQL 把 phone 转成数字）
WHERE phone = 13800138000

-- ✅ 走索引
WHERE phone = '13800138000'
```

## 三、联合索引设计原则

### 3.1 四大原则

| 原则 | 说明 | 示例 |
|---|---|---|
| **等值在前，范围在后** | 等值查询的列放前面，范围查询的列放最后 | `(a, b, c)` 中 c 是 `> / BETWEEN` → 放最后 |
| **区分度高的在前** | 选择性越大的列越靠前 | 身份证号放前，性别放后 |
| **考虑覆盖索引** | 高频查询的列直接包含在索引中 | `SELECT a,b,c` → 索引直接覆盖 |
| **不超过 3~5 列** | 索引列太多会降低写入性能 | 单索引列数控制在合理范围 |

### 3.2 索引"能干啥"总结

```sql
INDEX idx(a, b, c)

idx 相当于建了三个索引：
  idx(a)         ← 确定（因为 a 在最左边，全局有序）
  idx(a, b)      ← 确定（a 确定后 b 有序）
  idx(a, b, c)   ← 确定

不能替代：
  idx(b)         ← 没有 a，b 不是全局有序
  idx(c)         ← 同理
  idx(b, c)      ← 同理
```

### 3.3 为什么联合索引比多个单列索引好？

1. **一个联合`(a,b,c)` 等于三个索引** `(a) / (a,b) / (a,b,c)`，维护成本低
2. 多个单列索引可能触发 **index merge**（合并多个索引结果），性能通常不如单个联合索引
3. 联合索引更容易实现**覆盖索引**避免回表

## 四、EXPLAIN 关键字段速查

### 4.1 type 字段（访问类型，性能从好到差）

```
const > eq_ref > ref > range > index > ALL
  │      │       │      │       │      │
 主键   关联表   普通   范围    全索引   全表
 等值   主键    索引   扫描    扫描    扫描
 常量   等值
```

| type | 含义 | 示例 |
|---|---|---|
| **const** | 主键等值查询，最多返回一行 | `WHERE id = 1` |
| **eq_ref** | 关联表用主键/唯一索引关联 | `JOIN ON a.id = b.id` |
| **ref** | 普通索引等值查询 | `WHERE name = '张三'` |
| **range** | 索引范围扫描 | `WHERE id > 10 AND id < 100` |
| **index** | 全索引扫描 | `SELECT name FROM user`（name 有索引） |
| **ALL** | **全表扫描（最差）** | `WHERE age = 25`（age 无索引） |

### 4.2 Extra 关键值

| Extra 值 | 含义 |
|---|---|
| **Using index** | 覆盖索引，不回表（最优） |
| **Using index condition** | ICP 索引下推 |
| **Using where** | Server 层二次过滤 |
| **Using filesort** | 额外排序（需要优化） |
| **Using temporary** | 用临时表（最需要优化） |

## 五、终极背诵总结

1. **最左前缀**：联合索引按 a→b→c 排序，没有 a 就无法用索引
2. **失效十条**：%LIKE / 类型转换 / 函数 / 运算 / OR混合 / != / IS NULL / 断左前缀 / 优化器放弃 / ORDER BY 不连续
3. **设计原则**：等值在前范围在后 + 区分度高在前 + 覆盖索引优先 + 列数 3~5
4. **type 排序**：const → eq_ref → ref → range → index → ALL
5. **Extra 关注**：Using filesort / Using temporary 是红灯
