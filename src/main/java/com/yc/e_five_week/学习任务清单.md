## Day 1：InnoDB、MyISAM、行存储与页结构基本概念

### 📖 基础知识点
- [ ] 存储引擎对比：InnoDB（支持事务/行锁/MVCC/外键，默认引擎）vs MyISAM（表锁/不支持事务/不支持外键，适合读多写少）
- [ ] InnoDB 页结构：默认 16KB，页是 InnoDB 磁盘管理的最小单位，包含 File Header / Page Header / Infimum+Supremum Records / User Records / Free Space / Page Directory / File Trailer
- [ ] 行格式（Row Format）：Compact / Dynamic / Compressed / Redundant，影响行存储结构和溢出页处理
- [ ] InnoDB 表空间：系统表空间 / 独立表空间（`innodb_file_per_table=ON`）/ undo 表空间 / 临时表空间
- [ ] 内存结构：Buffer Pool（缓存数据页和索引页）→ Change Buffer（缓存非唯一二级索引的修改）→ Adaptive Hash Index（自动构建哈希索引）→ Log Buffer（redo log 缓冲）

### ❓ 常见面试题
**Q1: InnoDB 和 MyISAM 的核心区别？**
> 答：InnoDB 支持事务（ACID）、行级锁、MVCC、外键，适用 OLTP；MyISAM 不支持事务、表级锁，适用读多写少的场景（如数据仓库）。MySQL 5.5+ InnoDB 成为默认引擎。

## Day 2：B+ 树、聚簇索引、二级索引、回表、覆盖索引

### 📖 基础知识点
- [ ] B+ 树特点：① 只有叶子节点存数据，非叶子节点只存索引键（每页能存更多索引项，树更矮）；② 叶子节点形成有序双向链表（适合范围查询）
- [ ] 聚簇索引：数据即索引，叶子节点存完整行数据；InnoDB 一定会建聚簇索引（有主键用主键，无主键用第一个唯一非空索引，都没有生成隐藏 row_id）
- [ ] 二级索引：叶子节点存索引键+主键值（不是整个行数据）
- [ ] 回表：二级索引查到主键 → 回到聚簇索引取完整行（额外一次 B+ 树查找，性能损耗）
- [ ] 覆盖索引：查询的列都在二级索引中 → 不需要回表，`explain` 的 Extra 显示 `Using index`

### ❓ 常见面试题
**Q1: 为什么 MySQL 用 B+ 树而不用红黑树或 B 树？**
> 答：红黑树每个节点存一个键+数据，树高太高（数据量大时磁盘 IO 次数大，不适合磁盘存储）；B 树非叶子节点也存数据，同样高度下能存的索引项少，且叶子节点无链表不支持高效范围查询；B+ 树非叶子节点只存键（更高扇出→更矮），叶子有双向链表（范围查询连续读取），适合磁盘 IO。

**Q2: 什么是回表？怎么避免？**
> 答：二级索引找到主键后，再去聚簇索引查完整行数据的过程叫"回表"。避免方式：使用覆盖索引，让查询列都包含在二级索引中（Extra=Using index）。

## Day 3：最左前缀、索引失效场景、联合索引设计

### 📖 基础知识点
- [ ] 最左前缀规则：联合索引 (a, b, c) → 相当于建了 (a)、(a,b)、(a,b,c) 三个索引，跳过 a 直接用 b 或 c 无法走索引
- [ ] 索引失效 10 大场景：① like 以 % 开头（`%abc`）；② 隐式类型转换（`varchar = 数字`）；③ 索引列上使用函数（`WHERE DATE(create_time) = '2024-01-01'`）；④ 索引列上运算（`WHERE id + 1 = 10`）；⑤ OR 条件中有一个非索引列；⑥ `!=` / `<>` / `NOT IN` 在某些情况不走索引；⑦ `IS NULL` / `IS NOT NULL` 在某些情况不走；⑧ 联合索引不满足最左前缀；⑨ 全表扫描比索引更快时（优化器选择）；⑩ `ORDER BY` 不满足最左前缀
- [ ] 联合索引设计原则：等值查询放前面，范围查询放后面；区分度高的列放前面；考虑覆盖索引
- [ ] `explain` 关键字段：type（访问类型，const>eq_ref>ref>range>index>ALL）、key（实际使用的索引）、rows（预估扫描行数）、Extra（Using index/filesort/temporary）

### ❓ 常见面试题
**Q1: 什么是最左前缀？为什么会有这个规则？**
> 答：联合索引 (a,b,c) 按 a→b→c 顺序构建 B+ 树，a 先排序，a 相同再排 b，b 相同再排 c。所以查询时必须从 a 开始，否则索引无法定位范围。这是 B+ 树的物理结构决定的。

**Q2: 为什么建议建联合索引而不是多个单列索引？**
> 答：① 一个联合索引可满足多种查询（最左前缀）；② 多个单列索引可能导致 index merge，性能不如单个联合索引；③ 联合索引更容易实现覆盖索引，避免回表。

## Day 4：事务 ACID、隔离级别、脏读/不可重复读/幻读

### 📖 基础知识点
- [ ] ACID：原子性（undo log 回滚）、一致性（约束/触发器保证）、隔离性（锁+MVCC 保证）、持久性（redo log 保证 crash recovery）
- [ ] 4 种隔离级别：READ UNCOMMITTED（脏读）→ READ COMMITTED（不可重复读，每次读最新已提交版本）→ REPEATABLE READ（幻读，InnoDB 的 RR 通过 Next-Key Lock 解决了幻读）→ SERIALIZABLE（串行化，逐条执行）
- [ ] 3 种并发问题：脏读（读到未提交的数据）→ 不可重复读（同一事务两次读结果不同，数据被修改）→ 幻读（同一事务两次读结果不同，多了/少了行）
- [ ] InnoDB 默认 RR，Oracle 默认 RC

### ❓ 常见面试题
**Q1: MySQL 隔离级别是怎么实现的？**
> 答：READ UNCOMMITTED：读不加锁，看到最新（不管是否提交）；READ COMMITTED：每次读生成新的 ReadView，MVCC 读已提交版本；REPEATABLE READ：事务开始时生成 ReadView 且不变，配合 Next-Key Lock（行锁+间隙锁）解决当前读的幻读；SERIALIZABLE：所有读都加共享锁。

**Q2: InnoDB 的 RR 级别下幻读完全解决了吗？**
> 答：快照读（普通 SELECT）通过 MVCC 解决了幻读；当前读（SELECT ... FOR UPDATE）通过 Next-Key Lock 解决。但在极端场景（如先快照读再当前读更新）可能感知到幻读。

## Day 5：MVCC、undo log、redo log、binlog

### 📖 基础知识点
- [ ] MVCC（多版本并发控制）：每行记录有两个隐藏列 `trx_id`（最近修改的事务 ID）和 `roll_pointer`（回滚指针，指向 undo log 的上一个版本）
- [ ] ReadView：包含活跃事务列表，判断某个版本是否可见（`trx_id < min_trx_id` 可见，`trx_id > max_trx_id` 不可见，中间则判断是否在活跃列表中）
- [ ] undo log（回滚日志）：记录数据修改前的版本，用于事务回滚和 MVCC 读取旧版本（保证一致性读不阻塞写）
- [ ] redo log（重做日志）：物理日志，记录"数据页做了什么修改"，循环写入，用于宕机恢复（保证持久性），WAL（Write Ahead Log）策略
- [ ] binlog（归档日志）：MySQL Server 层逻辑日志，记录 SQL 语句/行变化，用于主从复制和基于时间点的恢复

### ❓ 常见面试题
**Q1: redo log 和 binlog 的区别？**
> 答：redo log 是 InnoDB 引擎层的物理日志，记录"某页某偏移量修改了什么"；binlog 是 Server 层的逻辑日志，记录 SQL 语句或行变更。redo log 循环写（不保存历史），用于崩溃恢复；binlog 追加写（保存历史），用于主从复制和恢复。两阶段提交保证一致性。

**Q2: MVCC 解决了什么问题？**
> 答：MVCC 让读不阻塞写、写不阻塞读。快照读（普通 SELECT）直接读历史版本，不需要加锁；写操作创建新版本，不影响正在读的旧版本。实现了高效的并发控制，尤其是 RR 级别下的一致读。

## Day 6：锁机制、行锁、间隙锁、Next-Key Lock、死锁与 explain

### 📖 基础知识点
- [ ] InnoDB 锁类型：共享锁（S 锁，读锁）→ `SELECT ... LOCK IN SHARE MODE`；排他锁（X 锁，写锁）→ `SELECT ... FOR UPDATE`
- [ ] 行锁粒度：Record Lock（锁住索引记录本身）、Gap Lock（锁住索引记录间的间隙，防止插入）、Next-Key Lock（= Record Lock + Gap Lock，InnoDB RR 下默认）
- [ ] 死锁发生条件：互斥+持有并等待+不可剥夺+循环等待；InnoDB 自动检测死锁（`innodb_deadlock_detect=ON`），回滚代价小的事务
- [ ] `explain` 实战：看 type（const/eq_ref/ref/range/index/all）、key、rows、Extra（Using index/Using filesort/Using temporary）
- [ ] `explain analyze`（MySQL 8.0.18+）：实际执行并返回每步耗时

### ❓ 常见面试题
**Q1: 什么是 Next-Key Lock？为什么需要它？**
> 答：Next-Key Lock = 行锁（Record Lock）+ 间隙锁（Gap Lock），锁住索引记录 + 它前面的间隙。用于解决 RR 级别下当前读的幻读问题。例如 `WHERE id > 10 AND id < 20 FOR UPDATE`，锁住 (10, 20] 的间隙和满足条件的行。

**Q2: 死锁怎么排查和避免？**
> 答：排查：`SHOW ENGINE INNODB STATUS` 找 `LATEST DETECTED DEADLOCK` 段，看等待链。避免：① 以相同顺序访问资源；② 尽量缩短事务；③ 合理使用索引减少锁范围；④ 必要时用 `SELECT ... FOR UPDATE NOWAIT/SKIP LOCKED`。
