今日作业 1：MQ 消息可靠性全链路保障清单
    1. 生产者端（保证消息不丢）
        确认机制：开启 publisher-confirm，Broker 收到消息后回调通知生产者
        事务 / 重试：发送失败时重试（设置重试次数 + 间隔），或用本地事务表保证消息发送成功
        幂等发送：每条消息带唯一 ID，避免重复发送
    2. Broker 端（保证消息不丢）
        持久化：队列、交换机、消息都设置持久化
        集群部署：主从复制 / 集群模式，避免单点故障
        镜像队列：RabbitMQ 配置镜像队列，消息同步到多个节点
    3. 消费者端（保证消息不丢、不重复）
        手动 ACK：消费成功后手动确认，失败时 nack 重回队列或转入死信队列
        幂等消费：用 messageId + Redis / 数据库去重，避免重复处理
        死信队列：处理失败次数过多的消息，转入死信队列人工排查
今日作业 2：消费堆积排查思路和解决步骤（口述版）
    1. 排查思路
        定位堆积原因：
        消费者是否宕机 / 重启？
        消费者处理速度是否跟不上生产速度？
        是否有消息处理异常，导致一直 nack 重回队列？
        是否网络 / 配置问题导致消费失败？
        定位堆积位置：
        Broker 端查看队列长度、消费者连接数
        查看消费者日志，是否有报错 / 死锁 / 阻塞
    2. 解决步骤
        紧急扩容：临时增加消费者实例数，提高消费能力（注意保证幂等）
        恢复故障消费者：重启 / 修复异常消费者，恢复消费
        异常消息处理：将处理失败的消息转入死信队列，避免阻塞正常消费
        优化消费逻辑：
        简化业务逻辑，减少单条消息处理时间
        批量消费（如 RabbitMQ prefetch 批量拉取）
        异步处理非核心逻辑
        限流 / 削峰：生产者限流，或用延迟队列 / 死信队列削峰
