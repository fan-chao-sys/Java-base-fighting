## 一、缓存三件套

| 问题 | 现象 | 原因 | 解决方案 |
|---|---|---|---|
| **穿透** | 查不存在的数据，请求穿过缓存打到 DB | 恶意攻击/业务查询不存在数据 | ① **布隆过滤器**前置过滤 ② 缓存空值设短过期 |
| **击穿** | 热点 key 过期瞬间大量请求打到 DB | 热点 key 过期 + 高并发 | ① **互斥锁**（只让一个线程查 DB）② 逻辑过期+异步刷新 |
| **雪崩** | 大量 key 同时过期 / Redis 宕机 | 过期时间相同 / 单点故障 | ① 过期时间加**随机值** ② 高可用（主从+哨兵）③ 限流降级 |

### 1.1 布隆过滤器

```
判断 key 是否存在：
  - 不存在 → 一定不存在（直接返回，不查 DB）
  - 存在   → 可能存在（有误判率，查缓存/DB 二次确认）

原理：多个哈希函数 + 位数组，有 1% 误判率但内存极省
```

## 二、双写一致性（Cache Aside 最常用）

### 2.1 Cache Aside 模式

```
读：先查缓存 → 命中返回 / miss 则查 DB → 写入缓存 → 返回
写：先更新 DB → 再删除缓存（不是更新缓存！）
```

**为什么是删缓存不是更新缓存？** 更新缓存可能覆盖并发写的结果，删缓存更安全。

**为什么先更新 DB 后删缓存？** 先删缓存再更新 DB 中间有窗口可能写入旧值。

### 2.2 延时双删（更保险）

```
① 删除缓存 → ② 更新 DB → ③ 等待 N 毫秒 → ④ 再次删除缓存
```

### 2.3 最终一致性方案

```
Canal 监听 MySQL binlog → 发送到 MQ → 消费者更新/删除 Redis 缓存
```

## 三、分布式锁

### 3.1 从手写到 Redisson

```bash
# 1. 加锁（原子操作）
SET lock_key unique_value NX PX 30000
# NX=不存在才设置，PX=超时30秒，unique_value=自己生成的UUID

# 2. 解锁（Lua 脚本保证原子性 —— 判断是自己的锁才删）
if redis.call("get", KEYS[1]) == ARGV[1] then
    return redis.call("del", KEYS[1])
else
    return 0
end
```

### 3.2 分布式锁四大坑及解决

| 坑 | 解决 |
|---|---|
| **锁超时释放**（业务没执行完锁过期了） | **Redisson WatchDog** 自动续期（每 10s 续 30s） |
| **误删锁**（A 的锁被 B 释放了） | value 设唯一标识，Lua 脚本校验后才删 |
| **单点故障**（Redis 挂了锁全丢） | **RedLock** 多节点加锁（N/2+1 个节点加锁成功才算） |
| **主从切换丢锁** | RedLock 缓解，更极致用 ZooKeeper / etcd |

## 四、终极背诵总结

1. **穿透**：查不存在的数据 → 布隆过滤器 + 缓存空值
2. **击穿**：热点 key 过期 → 互斥锁 + 逻辑永不过期
3. **雪崩**：大量 key 同时过期 → 过期加随机值 + 主从哨兵 + 限流降级
4. **Cache Aside**：先更新 DB → 再删除缓存（最常用）
5. **分布式锁三件套**：`SET NX PX` 加锁 → Lua 脚本解锁 → WatchDog 续期
6. **四大坑**：锁超时（续期）/ 误删（校验 value）/ 单点（RedLock）/ 主从丢锁
