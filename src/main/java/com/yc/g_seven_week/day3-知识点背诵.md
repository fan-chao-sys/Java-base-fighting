# Day3 MQ 场景、可靠性、顺序性、重复消费、积压处理 底层核心知识点背诵笔记 ⭐⭐⭐

---

## 一、MQ 三大场景

| 场景 | 作用 | 举例 |
|---|---|---|
| **异步** | 非核心逻辑异步化，提高响应速度 | 注册后发邮件/短信，主流程返回，邮件异步发 |
| **解耦** | 服务间不再直接调用，通过消息中转 | 订单服务 → MQ → 库存/物流/积分服务 |
| **削峰填谷** | 平滑突发流量，保护下游 | 秒杀请求写入 MQ → 后端匀速消费 |

---

## 二、消息不丢 —— 全链路保障

| 环节 | 怎么做 |
|---|---|
| **生产者** | **confirm 回调**（发送方确认模式）/ 事务消息，失败重试 |
| **Broker** | 持久化（同步刷盘 + 主从同步复制） |
| **消费者** | **手动 ack**（消费成功再提交 offset），严禁自动 ack |

```
生产者发送 → Broker 持久化 → 消费者处理 → 手动 ACK
    │              │               │
 confirm/重试   刷盘+同步复制     幂等处理
```

---

## 三、消息重复消费 —— 幂等方案

MQ 通常保证"至少一次投递"，**幂等必须由消费端做**。

| 方案 | 做法 |
|---|---|
| **数据库唯一索引** | 业务唯一 ID 作为唯一约束，insert 去重 |
| **Redis setnx** | `SET msg_id 1 NX` 消费前打标 |
| **数据库乐观锁** | `UPDATE ... SET status=1, version=version+1 WHERE version=#{oldVersion}` |
| **消费记录表** | 消费前查记录表，已处理则跳过 |

---

## 四、顺序消息

- **局部有序**：同一个业务 ID（如订单号）的消息路由到同一个 Queue/分区
- RocketMQ：`MessageQueueSelector` 按业务 ID 取模选择 Queue
- Kafka：按 key 哈希到同一 Partition

---

## 五、消费积压处理

| 原因 | 方案 |
|---|---|
| 消费者处理慢 | ① 增加消费者并行度 ② 批量消费 |
| 生产者发送太快 | ① 临时扩队列分区数 ② 积压消息转存后批量处理 |
| 下游依赖慢 | 单独处理慢的那部分 |

```
紧急处理：
  ① 临时扩分区（Partition/Queue）→ 增加消费者
  ② 消费者直接写新 Topic，老 Topic 暂存
  ③ 问题修复后用程序逐步回放老 Topic
```

## 六、常见 MQ 对比

| | RocketMQ | Kafka | RabbitMQ |
|---|---|---|---|
| **吞吐量** | 高 | **极高** | 中等 |
| **消息可靠性** | 极高（同步刷盘+复制） | 高 | 高 |
| **事务消息** | ✅ 原生支持 | ❌ | ❌ 靠插件 |
| **顺序消息** | ✅ 原生支持 | 单分区有序 | 单队列有序 |
| **适用** | 金融/电商等业务场景 | 大数据/日志 | 中小型通用 |

## 七、终极背诵总结

1. **MQ 三场景**：异步（提速）、解耦（松耦合）、削峰（护下游）
2. **消息不丢**：生产 confirm → Broker 持久化 → 消费手动 ack
3. **幂等方案**：唯一索引 / Redis setnx / 乐观锁 / 消费记录表
4. **顺序消息**：同一业务 ID → 同一 Queue → 局部有序
5. **积压处理**：扩分区 + 加消费者 + 暂存后回放
