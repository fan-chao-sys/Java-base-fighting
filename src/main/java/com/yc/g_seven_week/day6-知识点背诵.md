## 一、CAP 定理

```
        C（一致性）
         /\
        /  \
       /    \
      / CAP  \
     /  只能   \
    /   选两个   \
   /            \
  A（可用性）────── P（分区容错性）
```

**因为网络分区（P）不可避免，所以实际上是在 C 和 A 之间取舍。**

| 选择 | 代表系统 | 行为 |
|---|---|---|
| **CP**（一致+分区） | ZooKeeper、etcd | 分区时牺牲可用性，拒绝写入 |
| **AP**（可用+分区） | Eureka、Nacos | 分区时牺牲一致性，允许写入但可能不一致 |

---

## 二、BASE 理论

**BASE = AP 方案的实践指导**：

| 字母 | 含义 | 说明 |
|---|---|---|
| **BA** | Basically Available | **基本可用**：允许部分降级，不影响核心链路 |
| **S** | Soft state | **软状态**：允许系统存在中间状态 |
| **E** | Eventually consistent | **最终一致性**：不要求实时一致，但最终会一致 |

## 三、幂等设计

### 3.1 接口幂等

| 方案 | 做法 |
|---|---|
| **Token 机制** | 先请求 token → 提交时校验 + 删除 token → 重复请求无有效 token |
| **数据库唯一索引** | 业务唯一 ID + `UNIQUE INDEX`，第二次 insert 报错 |
| **乐观锁** | `version` 字段，`UPDATE ... SET version=version+1 WHERE version=#{old}` |
| **状态机** | 已处理的重复请求直接返回结果 |

### 3.2 消息幂等

| 方案 | 做法 |
|---|---|
| **唯一索引去重** | 消息 ID + `UNIQUE INDEX` |
| **Redis 消费标记** | `SET msg_id 1 NX EX 3600` |
| **消费记录表** | 先查记录表，存在则跳过 |

## 四、分布式事务方案

| 方案 | 原理 | 一致性 | 适用场景 |
|---|---|---|---|
| **2PC** | 协调者：第一阶段询问，第二阶段提交/回滚 | 强一致 | 性能差，不适合高并发 |
| **TCC** | Try（预留）→ Confirm（确认）→ Cancel（回滚） | 强一致 | 资金/库存扣减 |
| **可靠消息最终一致** | 本地事务 + 消息表 + MQ 发送 | 最终一致 | 异步解耦场景（最常用） |
| **Seata AT** | 自动生成反向 SQL，二阶段回滚时执行 | 接近一致 | 不想侵入业务代码 |
| **SAGA** | 长事务拆为多个短事务，每个有补偿操作 | 最终一致 | 长流程（如旅行预订） |

## 五、限流算法

| 算法 | 原理 | 优点 | 缺点 |
|---|---|---|---|
| **计数器** | 固定窗口计数 | 简单 | 窗口边界突刺 |
| **滑动窗口** | 细分窗口时间片 | 平滑 | 实现复杂 |
| **漏桶** | 恒定速率流出 | 保护下游稳定 | 无法应对突发 |
| **令牌桶** | 固定速率生产令牌，可累积 | 允许突发流量 | 实现稍复杂 |

**实际使用**：Sentinel / Guava RateLimiter 都是**令牌桶**变种。

### 5.1 令牌桶 vs 漏桶

```
漏桶：   入 ═══╗         令牌桶：  令牌→放入←生成(定时)
            ║                     ║
            ▼ 恒速流出             ▼ 有令牌就能出，可突发
```

## 六、终极背诵总结

1. **CAP**：P 必须选（网络分区不可避免），C 和 A 二选一 → ZK(CP) vs Eureka(AP)
2. **BASE**：AP 的实践方案，基本可用 + 软状态 + 最终一致
3. **幂等**：Token 机制 / 唯一索引 / 乐观锁 / 状态机
4. **分布式事务**：2PC(强)/TCC(强)/可靠消息(最终)/Seata AT(简化)/SAGA(长流程)
5. **限流**：令牌桶（常见，允许突发）> 漏桶（恒定）> 滑动窗口 > 计数器
