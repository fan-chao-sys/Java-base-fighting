## Day 1：Redis 数据类型、底层结构、持久化、过期和淘汰策略

### 📖 基础知识点
- [ ] 5 大数据类型及场景：String（缓存/计数/分布式锁）、Hash（对象存储）、List（消息队列/最新列表）、Set（去重/抽奖/共同好友）、ZSet（排行榜/延时队列）
- [ ] 底层数据结构：SDS（简单动态字符串）、双向链表、压缩列表（ziplist，小数据）、跳跃表（skiplist，ZSet/有序）、哈希表、整数集合（intset）
- [ ] RDB（快照）：`save`/`bgsave` 生成全量数据快照，恢复快但可能丢数据；AOF（追加日志）：每次写操作追加日志，更可靠但恢复慢、文件大
- [ ] 混合持久化（Redis 4.0+）：RDB + AOF 结合，既快又全
- [ ] 过期策略：惰性删除（访问时检查）+ 定期删除（每 100ms 抽样删除）
- [ ] 淘汰策略（8 种）：noeviction（不淘汰，满写报错）/ allkeys-lru / volatile-lru / allkeys-lfu / volatile-lfu / allkeys-random / volatile-random / volatile-ttl

### ❓ 常见面试题
**Q1: Redis 为什么快？**
> 答：① 纯内存操作（数据在内存中）；② 单线程模型（无多线程竞争和上下文切换开销，6.0+ 网络 IO 多线程但执行仍是单线程）；③ IO 多路复用（epoll）；④ 高效的数据结构（SDS/跳跃表/压缩列表都是为 Redis 定制的）；⑤ RESP 协议简单。

**Q2: RDB 和 AOF 持久化分别适合什么场景？**
> 答：RDB：子进程 fork 快照，对性能影响小，恢复快，适合备份和灾备，但可能丢最近几分钟的数据；AOF：记录每个写操作，更可靠，适合对数据安全要求高的场景，但文件大且恢复慢。生产环境建议两者都开。

## Day 2：缓存穿透、击穿、雪崩、双写一致性、分布式锁

### 📖 基础知识点
- [ ] 缓存穿透：查不存在的数据，缓存和 DB 都没有，请求打到 DB → 解决：布隆过滤器（提前过滤不存在 key）、缓存空值（设置短暂过期时间）
- [ ] 缓存击穿：热点 key 过期瞬间大量并发请求打到 DB → 解决：互斥锁（只有一个线程去查 DB 重建缓存）、逻辑过期（永不过期+异步刷新）
- [ ] 缓存雪崩：大量 key 同时过期或 Redis 宕机 → 解决：过期时间加随机值、高可用（主从+哨兵）、限流降级
- [ ] 双写一致性方案：① 先更新 DB 后删缓存（Cache Aside，最常用）；② 先删缓存再更新 DB（延时双删）；③ 读写串行化（MQ/Canal 订阅 binlog 更新缓存）
- [ ] 分布式锁：`SET key value NX PX timeout`（原子操作）+ Lua 脚本解锁（判断 value 是否匹配再删除，防止误删）+ Redisson WatchDog 自动续期

### ❓ 常见面试题
**Q1: Redis 分布式锁有什么坑？怎么解决？**
> 答：① 锁超时释放 → Redisson WatchDog 自动续期；② 误删锁（A 的锁被 B 释放）→ value 设为唯一标识（UUID+线程ID），解锁时用 Lua 脚本校验；③ 单点故障 → RedLock 多节点；④ 主从切换丢锁 → 也是 RedLock 缓解。

**Q2: 缓存和数据库双写一致性怎么处理？**
> 答：常用 Cache Aside 模式：先更新数据库，再删除缓存。读请求 miss 后查数据库并写入缓存。极端情况下短暂不一致可接受（如先 delete 缓存再更新 DB 中间有查询写入旧值）。强一致性可用 Canal + MQ 异步更新。

## Day 3：MQ 场景、可靠性、顺序性、重复消费、积压处理

### 📖 基础知识点
- [ ] MQ 三大场景：异步（提高响应速度）、解耦（服务间不再直接调用）、削峰填谷（平滑突发流量）
- [ ] 消息不丢三环节：生产者 → 确认机制（confirm/事务）/ 重试；MQ 自身 → 持久化+集群同步；消费者 → 手动 ack 确认消费完毕再提交
- [ ] 重复消费：消息可靠性保证是"至少投递一次"，消费者要做幂等（唯一 ID+数据库去重表/Redis setnx 标记）
- [ ] 顺序消息：单一分区/Queue 保证局部有序；RocketMQ 的 MessageQueueSelector 按业务 ID 路由
- [ ] 消费积压：① 增加消费者并行度；② 临时扩队列分区数；③ 批量消费；④ 积压消息转存后批量处理

### ❓ 常见面试题
**Q1: MQ 如何保证消息不丢？**
> 答：生产端：开启 confirm 回调 / 事务消息，发送失败重试；Broker 端：持久化（刷盘+主从同步）；消费端：手动 ack（消费成功再提交 offset），不要自动 ack。RocketMQ 的同步刷盘+同步复制可做到消息零丢失。

**Q2: 如何避免重复消费？**
> 答：消费者做幂等处理：① 业务唯一 ID + 数据库唯一索引（插入即去重）；② Redis setnx（消费完标记）；③ 数据库乐观锁（version 字段）；④ 消费前查状态表判断是否已处理。MQ 本身通常保证"至少一次投递"，无法保证"仅一次"，所以幂等必须在消费端做。

## Day 4：TCP、UDP、HTTP、HTTPS、三次握手、四次挥手

### 📖 基础知识点
- [ ] TCP vs UDP：TCP 面向连接/可靠/有序/流量控制/拥塞控制，适合文件传输、HTTP；UDP 无连接/不可靠/低延迟，适合视频直播、DNS
- [ ] 三次握手：① Client → Server：SYN（seq=x）② Server → Client：SYN+ACK（seq=y, ack=x+1）③ Client → Server：ACK（ack=y+1）。原因：防止旧连接请求到达服务端建立无效连接（已失效的 SYN 到了服务端不会有第三次确认）
- [ ] 四次挥手：① Client → Server：FIN ② Server → Client：ACK（服务端可能还有数据要发）③ Server → Client：FIN ④ Client → Server：ACK → 等待 2MSL（确保最后一个 ACK 到达+网络残留包消失）
- [ ] HTTP vs HTTPS：HTTPS = HTTP + TLS/SSL，多了身份认证、加密、数据完整性校验

### ❓ 常见面试题
**Q1: 为什么 TCP 建立连接是三次握手，断开是四次挥手？**
> 答：三次握手的原因：两次不可靠（服务端无法确认客户端收到了自己的 SYN-ACK），三次刚好能确认双方收发能力都正常。四次挥手的原因：TCP 全双工，`FIN` 只表示"我不发数据了"但还能接收，服务端可能还有数据要发，所以 ACK 和 FIN 分开发送，多一次。

**Q2: HTTPS 比 HTTP 多了什么？**
> 答：HTTPS = HTTP + TLS/SSL 层。多了：① 加密传输（对称+非对称混合加密）；② 身份认证（CA 证书验证服务端身份）；③ 数据完整性校验（MAC 消息认证码防篡改）。TLS 握手约增加 2 个 RTT。

## Day 5：Session、Cookie、Token、鉴权流程

### 📖 基础知识点
- [ ] Cookie：浏览器端存储，随每次请求自动发送，有大小限制（4KB）和数量限制
- [ ] Session：服务端存储，基于 SessionID（存在 Cookie 中）关联用户状态；缺点：占用服务端内存、分布式需共享
- [ ] Token（JWT）：无状态认证，服务端签名后发给客户端，客户端每次请求带在 Header 中；优势：不占服务端存储、适合分布式
- [ ] JWT 结构：Header（算法）+ Payload（用户信息+过期时间）+ Signature（签名防篡改）
- [ ] SSO 单点登录：CAS（中心认证服务）/ OAuth2.0 / JWT 跨域共享

### ❓ 常见面试题
**Q1: Cookie、Session、Token 的区别和适用场景？**
> 答：Cookie 存客户端，是小容量的本地存储；Session 存服务端，通过 Cookie 中的 SessionID 关联，适合传统 Web 应用；Token（JWT）无状态，服务端不存，适合分布式/微服务/移动端。JWT 的缺点是无法主动失效（除非配合黑名单）。

## Day 6：CAP、BASE、幂等、分布式事务、限流和熔断降级

### 📖 基础知识点
- [ ] CAP 定理：C（一致性）/ A（可用性）/ P（分区容错性）只能同时满足两个，因为网络分区不可避免（P 必须选），所以大多数系统在 C 和 A 间取舍（CP：ZooKeeper / AP：Eureka）
- [ ] BASE 理论：Basically Available（基本可用）+ Soft state（软状态）+ Eventually consistent（最终一致性），是 AP 的实践方案
- [ ] 幂等设计：① 数据库唯一索引；② Token 机制（请求前获取 token，提交时校验）；③ 乐观锁（version 字段）；④ 状态机（已处理的重复请求直接返回）
- [ ] 分布式事务方案：① 2PC/3PC（强一致，性能差）；② TCC（Try-Confirm-Cancel，补偿型）；③ 可靠消息最终一致性（本地消息表 / RocketMQ 事务消息）；④ Seata（AT/TCC/SAGA 多种模式）
- [ ] 限流算法：计数器（固定窗口，突刺问题）→ 滑动窗口 → 漏桶（恒定速率）→ 令牌桶（允许突发，常用，Guava RateLimiter / Sentinel）

### ❓ 常见面试题
**Q1: 幂等在接口和消息场景里怎么做？**
> 答：接口：① 前端防重（按钮禁用+loading）；② 后端 Token 机制（先获取 token 再提交，校验删除 token）；③ 唯一索引去重。消息：① 业务唯一 ID + 消费记录表；② Redis setnx 消费标记；③ 数据库唯一约束（insert 去重）。

**Q2: 限流常见算法有哪些？各有什么优缺点？**
> 答：计数器/滑动窗口：简单，但窗口边界有突刺问题；漏桶：恒定速率，适合保护下游，但无法应对突发；令牌桶：允许一定突发，最常用（Sentinel/Guava RateLimiter 都是令牌桶）。
