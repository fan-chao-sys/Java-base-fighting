## 二、性能优化案例模板

```
① 现象：
   "接口 A 在高峰期响应变慢，P99 从 100ms 飙升到 2s"

② 定位：
   "① 日志发现大量慢 SQL
    ② EXPLAIN 发现 type=ALL，全表扫描 50 万行
    ③ jstat 发现 Full GC 频繁，老年代持续占满"

③ 优化：
   "SQL 优化：建联合索引 idx(user_id, create_time)，type 从 ALL → range
    缓存优化：热点数据加 Redis 缓存，过期 5min + 随机 0~60s
    JVM 优化：-Xmn 调大年轻代，大对象直入老年代阈值调整"

④ 效果：
   "P99 从 2s → 80ms，QPS 从 200 → 800，Full GC 从 5 次/min → 0"
```

## 三、故障排查案例模板

```
① 故障现象：
   "凌晨 2:00 收到告警，订单接口大面积超时，错误率飙升到 30%"

② 排查过程：
   "① 看监控 → CPU 100%
    ② top -Hp → 找到 CPU 最高的线程
    ③ jstack → 发现大量线程 BLOCKED 在数据库连接获取
    ④ 查连接池 → 连接池耗尽（maxActive 设了 20）
    ⑤ 查慢查询 → 发现某条 SQL 走了全表扫描，拖慢所有连接"

③ 临时止血：
   "① 重启服务释放连接
    ② 把连接池 maxActive 临时调大到 50
    ③ 加了紧急索引"

④ 长期方案：
   "① 所有 SQL 上线前必须 EXPLAIN 校验
    ② 连接池 maxWait 设置超时 + 熔断
    ③ 加慢查询监控告警（> 500ms 钉钉告警）
    ④ 连接池指标接入 Grafana"
```

## 四、高频面试反问话术

### Q1: "你做过哪些性能优化？"
> "我对订单列表接口做了两次优化。第一次是 SQL 层面，EXPLAIN 发现走了全表扫描，建了一个联合索引 `(user_id, create_time)`，type 从 ALL 到 range，扫描行数从 50 万降到 1000 行，响应从 2s 降到 200ms。第二次是缓存层面，把用户维度的热点数据用 Redis 缓存，key 过期加随机值防雪崩，最终 P99 稳定在 50ms 内。"

### Q2: "遇到最有挑战的问题是什么？"
> "线上 Full GC 频繁导致接口间歇性超时。通过 jstat 确认了 FGC 频率，jmap dump 后用 MAT 分析发现是一个定时任务每次都创建大对象直入老年代。解决方案是优化对象生命周期 + 调大年轻代。这个过程中最大的挑战是复现和定位，因为只在高峰期间歇性出现。"
